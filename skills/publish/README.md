# publish

The publish-quality gate: the moment work leaves the machine — a repo going public, a release, a
plugin-directory submission, design cards — the publication surface (README, listing, notes) owes its
reader what the artifact's KIND owes.

- a **skill** shows install lines, the commands as typed, when to use it and when NOT;
- a **tool** shows at least one real run with its real output;
- a visual **product** shows FRESH screenshots — re-shot at every publish that changes them;
- **prose** shows its reading path.

Comparisons and diagrams join when they carry the argument, never as decoration. Each publish TARGET
(GitHub, a plugin directory, a design project) is a plugin embedding its own steps into the walk — and
none of it ever replaces the human's publish gate: the skill prepares the deposit, the human releases it.

Part of the [live-spec](https://github.com/happysasha18/live-spec) pack (the shared rules live in
`live-spec-base`); readable standalone as plain advice. See `SKILL.md` for the full checklist.
