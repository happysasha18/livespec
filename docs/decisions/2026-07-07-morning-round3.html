Образец A — «Утверждённый прототип — эталон вида» (раунд 4)

Feature: An approved prototype is the design reference

User Story: As the product owner, I want every feature that started from an approved prototype to be built and reviewed against that prototype, so that the shipped result looks the way I approved it.

You approve a sketch: "Yes, this is the door." The approved file is saved as the design reference. Whoever builds the feature opens that file first and builds what it shows. Review checks the finished work against the same file.

Acceptance Criteria:

  1. WHEN the owner approves a prototype THEN a copy of it SHALL be frozen under docs/norms/ with a short note: what it is, when it was approved, which sketch it came from.
  2. WHEN a requirement comes from an approved prototype THEN the requirement SHALL end with a link to the frozen copy (norm: docs/norms/door.html).
  3. Requirements SHALL link the frozen copy only. A working sketch can change or be deleted, so it is an invalid reference.
  4. WHEN a builder starts such a requirement THEN the builder SHALL open the reference first and write one line describing how the plan differs from it.
  5. IF the work was done without opening the reference THEN review SHALL reject it.
  6. The final check SHALL compare the shipped look and feel to the same reference.
  7. WHILE a task carries the condition "show me a mockup first", the condition SHALL stay in force until the owner cancels it by name. A general "go ahead and build" leaves it in force.

Why this exists: A gallery was once rebuilt from its text description alone. All 75 tests passed, because they had been written from the same misreading of the text. The shipped gallery looked nothing like the approved design. (tlvphoto, 2026-07-05) [INV-43]

Образец B — «Баг режет очередь» (раунд 4)

Feature: A bug takes the lane; parked work returns first

User Story: As the product owner, I want a bug in the shipped product fixed before anything else, and I want the interrupted feature to come back on its own afterwards, so that the product heals fast and no work gets lost.

You are mid-feature and you report: "The card is broken on the phone." The feature is parked with a checkpoint, the bug is fixed first, and the parked feature then returns as the next thing to finish.

Acceptance Criteria:

  1. WHEN a bug report arrives while a feature is in progress THEN the feature SHALL be parked with a checkpoint: the failing test names, the current hypothesis, the touched files.
  2. Unfinished work with failing tests SHALL never be committed.
  3. WHILE a bug fix is in progress, new bug reports SHALL wait in the queue, whatever their priority.
  4. WHEN several bugs are waiting THEN critical bugs SHALL go first. Bugs of equal priority go in arrival order.
  5. WHEN the last waiting bug is fixed THEN parked features SHALL resume ahead of the whole queue, including quick wins.
  6. IF several lanes were parked at once THEN each lane SHALL resume from its own checkpoint, in the order the lanes were due to finish.

At most one feature is parked per lane. [T-9, T-11, T-18]

Вопрос 2 — строгость критериев

Критерии приёмки можно писать двумя голосами. Строгий, как в образцах выше (стиль Kiro): «WHEN a bug arrives THEN the feature SHALL be parked…» — однозначный и машинно-проверяемый, звучит формально. Обычный (стиль BMAD): «A bug arriving parks the feature at a checkpoint.» — читается легче; проверяемость сохраняется, пока каждый пункт несёт одно поведение.
