## Findings

### F1 — INV-159's body claims three invariants cite it that do not

> "each self-enforcing landing rule that states its own instance and cites this law [INV-103, INV-134, INV-156, INV-160] — so a reader tracing 'binds forward' arrives at this one law, and each neighbour cites it in place of restating it." — PRODUCT_SPEC.md line 343 (INV-159 body)

INV-159's body asserts that INV-103, INV-134, INV-156, and INV-160 each "cite this law." Only INV-160
does (`binds forward [INV-159]`, line 659). INV-103 (lines 1509 / 1982), INV-134 (327 / 1833), and
INV-156 (442 / 2008) each state their own forward-binding instance and contain no `[INV-159]` anchor
(grep-confirmed: no INV-103/134/156 clause cites INV-159). The invariant's own Formal-index entry
(line 2011) omits the claim — it reads "each self-enforcing landing rule stating its own instance
[INV-103, INV-134, INV-156, INV-160]" with no "and cites this law" — so the body and the index of the
same invariant disagree. This is the same class of forward-binding citation inconsistency that
blocked row 322 in the first place (F1/F5, record `2026-07-15-322-...`: a "binds forward" claim that
does not resolve), now reintroduced in the reverse direction. A reader who trusts INV-159's body and
walks to INV-103/134/156 to confirm the citation finds none, and cannot tell whether the repointing is
incomplete or the claim is loose. The green suite missed it: `test_forward_binding_cites_are_repointed_off_the_silent_roots`
checks only that the plain-duty cites point AT INV-159, never that the invariants INV-159 says cite it
actually do.

Fold one of two ways. (a) Add `[INV-159]` to the forward-binding sentence of INV-103, INV-134, and
INV-156 (making the body's claim true); or (b) reword the body to match the index — drop "and cites
this law," since a self-enforcing landing rule states its own dated instance BY DESIGN (a mechanical
suite check reads the concrete cell) and legitimately restates rather than citing-in-place-of. Option
(b) is cleaner: these three restate by nature, and INV-160 — a class declaration, not a "landing
rule" — is the odd member of the listed group, so the "cites in place of restating" framing fits none
of them precisely. Either way, one edit closes it.

### F2 — `orphan_guard` misattributes a sibling suite that launches mid-block

> "It diffs this harness's live owner pids across entry and exit, so a concurrent sibling run (alive on entry) is ignored" — templates/headless_harness.py, `orphan_guard` docstring (line ~366)

`_surviving_owners` reads every harness-launched Chrome machine-wide (all `PROFILE_PREFIX` dirs with an
`OWNER_PID` marker, across every temp root), not only the ones THIS process started. `orphan_guard`
snapshots that machine-wide set at entry and flags any owner alive at exit whose pid was not in the
entry snapshot. A concurrent sibling suite (a second repo's browser run, a parallel CI job on one
runner) that launches a browser DURING the guarded block and is still alive at exit has a pid absent
from `before`, so the guard raises `AssertionError` blaming the current suite for an orphan it did not
create — a false red under parallel suites. The common single-suite case and any sibling already
running at entry are handled correctly; this is the narrow overlap window the entry-only diff cannot
close.

Since the pack ships no browser suite of its own, this bites only a consumer running overlapping
browser suites. Two options for the fold: (a) leave it and document the constraint in the docstring
("run `orphan_guard` only when no sibling harness suite starts mid-block"); or (b) scope the census to
pids THIS process launched by intersecting with the in-process `_LIVE_BROWSERS` registry's pids — at
the cost of the "by-deed, independent of the teardown code" property the helper is built to have.
Given the by-deed independence is the whole point, (a) is the lighter call; surface it to the human as
a taste decision.

`recommendation · concurrency (composition)`

### F3 — INV-160's parity casts INV-77's never-green manual row as a run-reddening net

> "each names the net that reddens a run on its own violation, past merely naming the fix. INV-77, INV-78, and INV-79 make the assertion shape itself the net — the real-device walk row the suite can never green" — PRODUCT_SPEC.md line 659 (INV-160 body)

The parity is stated uniformly — every member "names the net that reddens a run on its own violation."
Eight members fit cleanly: an automated assertion that fails the run on the violated fact. INV-77's
net, though, is "the real-device walk row the suite can never green" — a permanently non-green manual
row owed to the human's hands, which does not mechanically redden a passing run on a violation the way
an automated assertion does; it is simply never asserted by the machine. INV-160's own text is honest
about this ("the suite can never green"), so this is not a false claim — a never-green row is arguably
"red" in the suite's ledger — but the uniform "reddens a run" phrasing papers over a real difference in
kind between an automated net and a manual owed-walk row. This is a same-kind-behaviour nuance (the
design-review register) rather than a spec hole.

Optional sharpening for the fold: in INV-160's parity sentence, distinguish INV-77's net as "the
never-green real-device walk row owed to the human's hands" from the automated assertions, so an author
adding a tenth member reads the parity as "names its net — an automated assertion that reds, or a
never-green owed-walk row," not "reddens a run" uniformly. No behaviour changes; the enumeration test
already passes.

`recommendation · over-general (abstraction)`
