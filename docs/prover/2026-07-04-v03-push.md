## Opening assessment

The v0.3 additions are sound in shape: write-ownership is a crisp yes/no test, the fence is mechanical,
and the inbox turns a shared-file race into conflict-free file creation. What needed attention was the
seams: the gate could regress on itself (fold → changed spec → re-prove forever?), an outsider's inbox
file interacts with the fence it was invented to satisfy, and nothing guaranteed an inbox wish is ever
NOTICED. All findable, all small. Ready to push after the folds below.

## Findings

F1 — Push-gate can chase its own tail (fold changes the spec that was just proven)
> "a fresh whole-spec re-check … Findings that are must-fix fold before pushing" — Milestones, M-6
The gate as written implies: prove → fold → the spec changed → prove again, with no stated floor.
An agent following it literally either loops or silently decides when to stop — a silent micro-decision
(INV-5 violation) inside the gate meant to prevent them.
Fold: one sentence in M-6 — folds produced by the gate's own pass do not re-trigger the gate; they ship
with the same record.
`must-fix · stuck-state (liveness)` → **FOLDED** (M-6)

F2 — Outsider's inbox file: committed or not? Either answer breaks something unless stated
> "creating a fresh file cannot collide" — Entities, E-11 · "if … the tree holds changes it did not make, it STOPS" — INV-11
If the outside session leaves its inbox file uncommitted, the next livespec session's fence sees foreign
uncommitted changes and stops — the mechanism designed to receive outsiders trips the alarm designed to
catch them. If it commits, the spec never said a commit was inside the read-only exception.
Fold: E-11 — the outsider commits its one new file, commit touching inbox/ only; INV-11 — new files under
inbox/ are the expected benign case, not a fence trip.
`must-fix · internal-conflict (consistency)` → **FOLDED** (E-11, INV-11)

F3 — An inbox wish can wait forever (nothing sweeps the inbox)
> "the next livespec session harvests inbox files into queue rows" — Entities, E-11
No trigger makes that session look. A host run drops a defect report; no livespec session runs for weeks;
the wish is durably recorded and operationally invisible — "no wish is lost" (INV-1) holds on disk and
fails in practice.
Fold: T-10 — a livespec session sweeps inbox/ as its FIRST act; M-1 — the milestone re-listing of open
human gates includes unharvested inbox files.
`should-clarify · stuck-state (liveness)` → **FOLDED** (T-10, M-1)

F4 — "cannot collide" is overstated on a name collision
> "One NEW file per wish (`YYYY-MM-DD-<source>-<slug>.md`) … cannot collide" — Entities, E-11
Same day, same source, same slug (two "fix-adopt" wishes) → the second writer must not touch the first
file, and the spec gives no rule.
Fold: E-11 — on an existing name, append `-2`, `-3`, ….
`worth-considering · missing-scenario (state-space)` → **FOLDED** (E-11)

F5 — Two prover records on one date have one name
> "its record landing in docs/prover/ before the push" — Milestones, M-6
Two pushes in a day → the second record overwrites or displaces the first (this very file is the live
example: 2026-07-04.md already existed).
Fold: M-6 — record name is `YYYY-MM-DD[-suffix].md`, suffix mandatory when the date file exists.
`worth-considering · missing-rule (invariant)` → **FOLDED** (M-6)

F6 — "assigned in this conversation" excludes standing maintenance routines
> "the human asked ME, in this conversation, to change livespec" — Invariants, INV-10
A scheduled/cron session the human explicitly created for livespec maintenance fails the literal test and
could refuse legitimate work (or worse, improvise its own justification).
Fold: INV-10 — a standing routine the human created FOR livespec counts as assignment.
`worth-considering · unclear-owner (actors)` → **FOLDED** (INV-10)

F7 — Fence and ownership are hand-discipline until the guardrails scaffold lands
> "Guardrails [target] — the mechanical checks wired to the pre-push hook" — Entities, E-6
Consistent with M-4's "discipline followed by hand and says so" — recorded here so row 3's acceptance
inherits it: the scaffold should mechanize the fence (HEAD check) and the push gate (prover-record-exists
check), not only the four listed checks.
`worth-considering · unenforceable-promise (discharge)` → queue note on row 3 (no spec change)
