### F1 — INV-157's no-orphan goal is unreachable on the abnormal-exit path it never states

> "On teardown it reaps the whole process group of the browser it launched … so a run leaves no orphan alive to accumulate across runs and saturate the machine" — PRODUCT_SPEC.md line 653 (INV-157)

The invariant's own reason to exist is that no orphan accumulates across runs. Its mechanism is a
teardown reap. But teardown runs only on a normal exit. Three reachable events skip it: the browser
subprocess crashes, the human interrupts the suite (Ctrl-C), or the machine sleeps mid-run — a case
the spec elsewhere names explicitly ("a sleeping machine kills them mid-write" [INV-95]). On any of
them the process group is never reaped, the helper/renderer/gpu children survive, and the next run
launches a fresh group beside the orphaned one — orphans accumulate across runs exactly as the
invariant forbids. This is the automated-path atomicity gap the prover treats as a defect: a browser
harness runs constantly and Chrome subprocess crashes are ordinary, and the harm (a saturated
machine) is the precise thing INV-157 exists to prevent. By the spec's own standard a reachable
situation with a blank answer is a finding of the fact-no-node-owns class [INV-72], and the delta is
meticulous about crash-safety everywhere else (a crash between commit and record still surfaces
exactly once [INV-146]; the walk's baseline restore point [INV-92]; the worker liveness verdict
[INV-76]) — the harness reap is the one place the abnormal exit goes unwritten. M-304 tests only the
teardown reap, so a green suite claims nothing about the crash path.

Add one clause to INV-157: the harness reaps not only its own teardown but any stale process group it
finds from a prior run at launch (a launch-time sweep keyed to the harness's own marker), or state by
name that the crash path is out of scope and owned elsewhere. The launch-time sweep is the derivable
answer — it is the same shape as INV-100's session-scoped before/after temp-home diff that already
catches a leaked FILE a crashed run left behind; the orphan PROCESS wants the same treatment.

`defect · partial-success-risk (atomicity)`

### F2 — the harness invariants omit the "ships the law, products instantiate" split their whole family states

> "A harness that drives a real browser starts it muted … so a test run makes no sound on the machine it runs on" — PRODUCT_SPEC.md line 653 (INV-157)

INV-157 and INV-158 are the newest members of a well-established family: a law about a user-/browser-
facing surface that live-spec itself cannot exercise because it has no such surface, so the pack ships
the rule and the template and leaves the real run to the products it serves. Every prior member states
that split in its own text: INV-125 ("this pack, a skill with no DOM, ships the rule and the prover
lens and leaves the DOM-wide assertion to the products it serves"), INV-136 ("live-spec itself has no
UI, so it ships the law and the frontend starter set and leaves the pixel assertion to the products it
serves"), INV-139 ("the browser-computed assertion living in the adopting project's suite (the INV-136
split, live-spec has no UI)"). INV-157/158 state no such split. The reader cannot tell from the spec
whether live-spec's own suite must launch a muted real browser (it does not — M-304/M-305 test the
template's CONTENT, `test_template_launches_muted` reading the shipped file for `--mute-audio`, not
driving Chrome) or whether the invariant binds only consuming products. The two delta invariants also
disagree with each other on enforcement locus by omission: INV-157 lists "a matrix row" as a home
(reading as own-suite enforcement), INV-158 lists the adopt path (reading as consumer-side), with
neither stating the split that reconciles them.

Add the family's split sentence to INV-157 and INV-158, in the exact shape INV-136/INV-139 use:
live-spec ships the harness law and the template and tests the template's content, and the real muted
browser run lives in the adopting project's own suite. This makes the enforcement locus explicit and
puts the two invariants back in sync with their siblings.

`recommendation · internal-conflict (consistency)`

### F4 — INV-158 homes adoption at a docs/records walk, but the harness is a code template a host extends

> "A consumer adopts it by updating the pack — the catch-up walk that brings a package update onto a host [INV-110] — and layers its own project-specific driving methods on the shared core" — PRODUCT_SPEC.md line 655 (INV-158)

The CROSS-LINK's F1 repointed INV-158's adopt home from the nonexistent build-pipeline path to the
catch-up walk [INV-110], which is right in spirit but lands on a walk whose own scope is narrower than
the case: INV-110's trigger is "the owner's ask to bring an adopted host's documentation or records
onto the current package," and it explicitly "does not fire on … a restructure of the host's own
product." The harness is not documentation — it is a code template the host layers its own driving
methods onto (INV-158's own words). Updating a shared code core beneath a host's project-specific
layer is a merge of code, not of records, so the plain catch-up walk does not by itself describe how
the host pulls the new core and re-applies its methods without clobbering either side. The mechanism
that actually carries a code change to hosts is INV-91's dated migration chapter (the release "owes
its hosts actions" and "ships a dated migration chapter stating the host-side steps"), which the
catch-up walk then executes — but INV-158 cites INV-110 alone and leaves INV-91 implicit.

Cite INV-91 beside INV-110 in INV-158: a harness-core change ships a migration chapter [INV-91] whose
steps the catch-up walk [INV-110] executes, so the code-update-and-re-layer path is explicit rather
than folded into a docs/records walk that disclaims host-product restructures.

`recommendation · boundary-issue (composition)`
