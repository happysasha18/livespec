## What the delta does (model)

`_sweep_stale_profiles` runs at every `Browser.__init__` **before** the run's own `mkdtemp` (harness
line 529 → 533). It globs `PROFILE_PREFIX + "*"` across every temp root and classifies each dir by its
`OWNER_PID` marker (`"<pid>\n<boot id>"`):

- **Owned, same boot, owner alive** → live concurrent run → skip (never touched).
- **Owned, same boot, owner dead** → crash leftover → `killpg` + `rmtree`.
- **Owned, cross/unknown boot** → pid meaningless → `rmtree` only, never signalled.
- **Ownerless (no marker / unparseable)** → NEW: reap **only if** `time.time() - getmtime(path) >
  OWNERLESS_STALE_AGE` (3600s); a young ownerless dir is left alone. `rmtree` only (no pid to signal).

Then a glut warning to stderr when `len(candidates) > PROFILE_GLUT_WARN` (50).

The owner marker is written at harness line 558–560, **inside `contextlib.suppress(Exception)`** — a
best-effort write that silently swallows any failure — a few milliseconds after `mkdtemp` (line 533) and
`Popen` (line 546).

### F1 — An old ownerless dir CAN belong to a live run, so the age reap can `rmtree` a profile Chrome is using right now

> "an old ownerless dir a killed run left before it ever recorded an owner (a young ownerless dir is left
> alone, since it may be a live sibling mid-launch)" — PRODUCT_SPEC.md, INV-157

The spec states an OLD ownerless dir *is* a killed run's leftover. That classification is falsifiable. The
owner marker at harness line 558 is written under `contextlib.suppress(Exception)`, so a failed write
leaves a **live** run's dir permanently ownerless. The write's most likely failure is `ENOSPC` — a full
temp — which is exactly the condition ROADMAP 333 targets (the 78 GB `$TMPDIR` leak). So under the
feature's own trigger condition, a live run's owner marker can silently fail to land; the dir is then
classified only by age. A long-lived run (the incident describes long sessions) that passes 3600s while
its profile-root mtime has frozen — plausible on macOS, where Chrome's post-launch churn is inside
`Default/` and cache subdirs, not the profile ROOT whose mtime `getmtime` reads — is then reaped by a
concurrent sibling's launch sweep. `shutil.rmtree` unlinks the live Chrome's user-data-dir; subsequent
Chrome reads/writes of the path fail and the live run dies mid-suite, reading as a product red — the very
false-red this change exists to end, reintroduced through the delete path. This also violates INV-162
("never a shared resource another party is using right now") and INV-157 ("never a live sibling").

The mtime-refresh mitigation the code leans on is unstated and fragile: it protects a live dir only while
top-level entries churn in the profile ROOT, and only matters at all in the write-failure case (a
successfully-marked dir never enters the ownerless branch). The whole false reap hinges on the suppressed
marker write.

Fix — one of, preference (a): **(a)** write an owner marker atomically right after `mkdtemp`, before
`Popen`, seeded with the harness's own (Python) pid, then overwrite with Chrome's pid — a live run then
never has an ownerless window, and the ownerless-age branch only ever sees truly dead litter; **(b)**
before `rmtree` of an old ownerless dir, confirm no live process holds it (a `lsof`-style open-handle
check or a scan for a running `--user-data-dir=<path>` process), matching INV-162's stated "an open
handle … checked before touching"; **(c)** stop suppressing `ENOSPC` on the marker write so an unmarked
live dir cannot arise silently. Age alone is a statistical proxy for "not in use," not the provable-dead
test INV-157's owned branch uses.

`defect · missing-rule (invariant)`

### F2 — The delete-safe discrimination is proven only by string-greps, not by deed

> "assert \"OWNERLESS_STALE_AGE\" in sweep … assert \"getmtime\" in sweep … assert \"rmtree\" in sweep" —
> tests/test_harness_template.py, `test_sweep_reaps_old_ownerless_litter_by_age`

The three new tests assert the code CONTAINS tokens; none exercises the behavior. No test creates an old
ownerless dir (backdated mtime) and asserts it is reaped, a young one and asserts it survives, or a
live-owned one and asserts it is untouched. A wrong age bound, an inverted comparison, or a false reap of
a live dir all pass green. This is exactly the class the 1.8.0 orphan-net caught only BY DEED (its record:
"the first cut false-reds when a sibling process launches mid-window — folded by the process-scope"). For
a directory-DELETING operation the by-deed gap matters most, and neither net covers it: the pack-side net
is a string check, and the consumer's by-deed net (`orphan_guard`/`surviving_orphans`) checks orphan
PROCESSES, not temp-dir reaping. INV-157 still "names a net," so INV-160's parity is not broken — but the
new reap sub-behavior's only net cannot catch its safety-critical failure.

Add a pack-side behavioral test: build a throwaway temp root, create an old ownerless dir (`os.utime` to
backdate past `OWNERLESS_STALE_AGE`), a young ownerless dir, and a live-owned dir (marker with
`os.getpid()`); call `_sweep_stale_profiles`; assert old gone, young present, live present. This directly
guards F1's fix.

`recommendation · now · missing-outcome-check (postcondition)`

### F3 — The glut warning counts dirs it did not sweep, and false-fires under legitimate parallelism

> "if len(candidates) > PROFILE_GLUT_WARN:" — templates/headless_harness.py, `_sweep_stale_profiles`

`candidates` is the pre-sweep glob count — every prefix-matching dir under the temp roots, INCLUDING live
concurrent siblings (skipped, never swept) and young ownerless dirs (left alone). Pre-sweep is the right
quantity to MEASURE the leak (post-sweep would miss the litter just cleaned), but the message —
"%d stale profile dirs … Swept this launch" — over-claims: a live sibling is counted as "stale" and
described as "swept" when it was deliberately left. A legitimately parallel run with more than 50
concurrent live browsers trips "the temp home may be filling … a run is leaking" at launch when nothing is
wrong. It is a stderr line and blocks nothing, so this is a taste call, not a blocker.

Warn on the count actually reaped this launch (increment a counter in each reap branch), or reword to
"%d harness profile dirs present under the temp roots at launch" and drop the "stale / Swept" claim so a
healthy parallel run is not told it is leaking.

`recommendation · later · hard-to-monitor (observability)`

### F4 — INV-162's "unprovable ownership → left untouched" reads against the ownerless-age reap; the age-as-non-use bridge is unstated

> "a resource in use, or one whose ownership the cleanup cannot prove, is left untouched" — PRODUCT_SPEC.md,
> INV-162

INV-162 (landed one version prior, v1.10.0) enumerates what a cleanup may touch: "what THIS run provably
created and owns, or a prior run of the same harness whose recorded owner is provably dead [INV-157]." An
ownerless dir has NO recorded owner, so it fits neither arm, and INV-162 elsewhere says a resource whose
ownership "the cleanup cannot prove" is left untouched. The new age reap touches exactly such a dir. The
same invariant also says the delete net touches "only a dead or **unowned** resource," which sanctions it
— so INV-162 reads two ways on the unowned case. The reap resolves the tension with age-as-proof-of-
not-in-use, but neither INV-157 nor INV-162 states age as the discriminator, and INV-162's own "current
use" test names "an open handle … checked before touching," which the age reap does not do. A reader
reconciling the harness against INV-162 finds the ownerless-age reap unbacked by the invariant's
enumerated whitelist.

State the bridge explicitly: add to INV-157/INV-162 that a prefix-matched harness dir with no owner marker
and an mtime older than the mid-launch window is provably not in current use and is a sanctioned third
reap basis, and reconcile INV-162's "unprovable ownership → untouched" (which should scope to
resources NOT provably created by this harness family — the human's browser, a foreign process) against
"only a dead or unowned resource" so the two clauses no longer read as a contradiction. Best paired with
F1(b) so the stated "open handle checked before touching" is actually true of the code.

`recommendation · now · internal-conflict (consistency)`

## Consistency checks that HOLD (no finding)

- **INV-100 ↔ INV-157 one-home.** INV-100's new sentence points at INV-157 for the launch sweep and
  states the OS-not-self-purging rationale; INV-157 owns the mechanism. No duplication, no contradiction.
- **INV-160 net-parity intact.** Members INV-77/78/79/80/100/102/155/157/158 unchanged; the sharpening
  adds no member and removes none. INV-100's net (temp-home before/after diff) and INV-157's nets
  (template string check + consumer by-deed process-group diff) are both still stated. The parity holds
  as written. (F2 notes the new reap sub-behavior lacks a by-deed net, but INV-157 still names a net, so
  the class clause is not falsified.)
- **Traceability (M-304).** The three new tests — `test_sweep_reaps_old_ownerless_litter_by_age`,
  `test_harness_warns_on_a_temp_glut`, `test_spec_states_the_temp_is_not_self_purging` — are all appended
  to M-304's INV-157 row and all present in `tests/test_harness_template.py`. No new anchor minted;
  correct for a PATCH (a sharpening of two existing invariants, no new invariant). Holds.
- **3600s floor vs the mkdtemp→pid-write window.** The normal window is `mkdtemp` → `Popen` → marker
  write = milliseconds (harness lines 533 → 546 → 558). 3600s is six-plus orders of magnitude above it, so
  a slow launch never crosses it. The floor is safe against launch latency; it is UNSAFE only against a
  suppressed marker write (F1) — a different failure mode than the one the floor was sized for.
- **TOCTOU within the sweep.** For an owned dir the live-owner check gates the reap; for an ownerless dir,
  a sibling "mid-launch" for over 3600s is not credible (launch is ms), so no realistic window for a dir
  to gain a live owner between the getmtime check and the rmtree.
