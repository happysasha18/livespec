# product-prover — FULL pass over SPEC.md v0.1 (2026-07-04, pre-publish gate)

TRIAGE: PROCEED — the doc describes a system with entities, a wish lifecycle, two entry modes, actors and
invariants. One standing caveat: it describes a PARTIALLY shipped system (skeleton exists; guardrails,
snapshots, router do not) with no per-claim reconciliation — findings F4 addresses this directly.

## Opening assessment

This spec turns a working discipline into a package other projects can attach to; its spine — the wish
lifecycle plus two entry modes — is the right shape, and the adoption sequence (orient → re-engineer →
attic → VCS gate → baseline) is unusually complete for a v0.1. Two things work well: every open decision
carries a current pick (no naked TBDs), and the provenance axis for adopted documents is a genuinely novel,
testable idea. Two things need attention before publish: the wish lifecycle has no exit except success —
yet the package's own queue already uses statuses ("closed", "waiting") the spec never defines; and the
spec claims disciplines its own repo does not yet instantiate, with no current-vs-target marking. Verdict:
needs one focused iteration — foldable today; ready to publish after the folds.

## Phase 1 — model (prose)

Entities: Host project (owns SPEC/matrix/queue/journal) · Wish · Queue · Spec · Matrix · Guardrails (4
checks) · Snapshot · User profile · Attic. Wish lifecycle: arrived → classified → spec-delta → validated →
queued → in-work → landed → reported. Actors: human (gates), senior agent (judgment), tiered workers
(mechanics). Entry modes: bootstrap (B-1), adopt (A-1..A-7). Assumed: single human per host; "the package
is a host of itself" (M-4) read as normative, not yet factual; snapshots read as per-surface, not
whole-artifact blobs.

## Findings

F1 — A wish can only succeed; the spec's own queue already disproves that
> "arrived → classified → … → landed → reported" — The life of a wish
ROADMAP.md rows 1 and 6 already carry statuses "waiting" and "closed" that map to no state here. The human
declines a wish, defers one, or a wish is absorbed by another — the operator then invents ad-hoc statuses
per row and INV-1's "no wish lost" becomes uncheckable because lost-vs-closed is undefined.
Add exit states: declined (human's word) · deferred (parked, revisit trigger named) · superseded (absorbed,
points to the absorbing row). Wishes are never DELETED — closed rows stay in the table.
`must-fix · missing-scenario (state-space)` → **FOLDED**

F2 — Bug preemption is named but the preempted wish's fate isn't
> "bugs may preempt the lane" — classified state
Mid-landing preemption arrives while the suite is red: the worker stops, the half-done work sits in the
tree, and after the bug lands the next session cold-starts with no defined resume point — the exact
restart-not-resume failure the checkpoint rule exists to prevent.
Define preemption: in-work → parked (checkpoint written: failing test names + hypothesis + touched files,
per the red-at-pause rule; nothing committed red) → resumes as the immediate next landing after the bug.
`must-fix · undefined-path (transitions)` → **FOLDED**

F3 — The completeness guardrail scans against a registry the spec never defines
> "the mechanical checks … completeness" — E-6
Completeness = "no user-facing surface empty or unregistered" — but there is no Surface-registry entity, so
an adopting project has nothing to instantiate and the check silently degrades to a hand-list, which is the
drift this package exists to kill.
Add entity: Surface registry — one named list per host of every user-facing surface; the completeness check
scans the real artifact against it; a rendered-but-unregistered surface is RED (self-closing).
`must-fix · missing-prerequisite (precondition)` → **FOLDED**

F4 — The spec promises disciplines its own repo doesn't yet run
> "this spec, this queue, these rules govern livespec's own development" — M-4
This very landing (SPEC v0.1) passed no guardrail, cited no matrix row, ran no test — a reader adopting the
package discovers the flagship repo doesn't do what the README sells, which reads as broken promise rather
than roadmap.
Mark current-vs-target explicitly: one status line up top naming which sections are shipped (templates,
adopt procedure text, skills) and which are target until their ROADMAP row lands (guardrails E-6, snapshot
E-7, router in ACT-3, M-4 enforcement).
`must-fix · unenforceable-promise (discharge)` → **FOLDED**

F5 — The user profile has no home and no owner for its most dangerous field
> "proactivity mode … trust level" — E-8
Nothing says where the profile lives or who may raise trust; an agent that raises its own trust level has
quietly widened its own permissions — the exact self-granted drift the method forbids.
Pin: profile lives at `.livespec/profile.md` in the host (global default fallback); mode and trust are
written ONLY on the human's word; the agent may propose, never set.
`should-clarify · unclear-owner (actors)` → **FOLDED**

F6 — Re-engineered claims are "unverified until reconciled" with no reconciling transition
> "marked unverified until reconciled" — A-3
An adopted spec claim can stay unverified forever while matrix rows get derived from it — tests then assert
folklore from the host's old docs.
Add the transition: an unverified claim is reconciled (file:line or removed) at the FIRST landing that
touches its surface, and all remaining ones at the first milestone — whichever comes first.
`should-clarify · undefined-path (transitions)` → **FOLDED**

F7 — When the diff baseline advances is unstated
> "the baseline the next run is diffed against" — E-7
If the baseline advances before the human-gated show at a milestone, an unwanted change becomes the new
"normal" and the diff guardrail goes blind to it.
State: the baseline advances only at landed, and only for the surfaces the change DECLARED; undeclared
surfaces keep the old baseline (that asymmetry is the whole trick).
`should-clarify · missing-outcome-check (postcondition)` → **FOLDED**

F8 — Two invariants live outside the invariants' home
> "[A-4] [INV-7]" / "[A-5] [INV-8]" — adopt steps 4–5
The package's own conflicts-guardrail (id defined in two homes / found in none) would flag its own spec —
INV-7 and INV-8 exist only as inline tags.
Move their one-line definitions into the Invariants block; keep the inline references.
`should-clarify · internal-conflict (consistency)` → **FOLDED**

F9 — A human-gated wish can wait forever with no re-surface rule
> "queued, needs Alexander" — ROADMAP rows 1, 4 pattern
INV-4 keeps the LANE moving, but the gated wish itself has no liveness: three weeks later nobody remembers
row 4 exists.
Add: every milestone report re-lists all open human gates (one line each).
`worth-considering · stuck-state (liveness)` → **FOLDED**

F10 — Skills are shared mutable state across hosts
> "Skills land in ~/.claude/skills/, available in every project" — README / install
Host A mid-migration under spec-author vN while the package ships vN+1 globally: host A's next session runs
different authoring rules mid-flight with no record of the switch.
Record the installed skills' versions in the host's `.livespec/` at attach time; a version change gets a
one-line journal note at next session start.
`worth-considering · persistence-drift (persistence and versions)` → **FOLDED**

F11 — Worker checkpoints default to /tmp, which a reboot erases
> (practice, not yet in the doc — location not clearly anchored)
A cut-off worker after a reboot has no checkpoint to resume from; the brief promised resumability.
Name the checkpoint home: host's `.livespec/checkpoints/` (gitignored), not /tmp.
`worth-considering · hard-to-operate (ops-ux)` → **FOLDED**

Coverage tables: authorization — N/A (single-user local tooling; security/privacy named out of scope:
nothing leaves the machine except explicit publish gates). CRUD — the only user-mutated persistent entity
is the queue row: create=intake · read=queue/report · update=status transitions (now incl. F1 exits) ·
delete=NEVER (closed rows persist). Invariants per state are stated for in-work and landed; arrived/
classified now inherit INV-1 explicitly (F1 fold).
