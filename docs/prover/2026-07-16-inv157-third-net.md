### F1 — The third net names a mechanical gate that does not exist

> "a guardrail scans every tracked script and reds a line that launches a real browser without the mute flag, so an unmuted launch anywhere in a consuming tree fails the run by machine" — INV-157, PRODUCT_SPEC.md:656

The clause is present tense — a shipped fact — but no such guardrail exists. `guardrails/` holds `check-broad-kill.sh` (INV-162's net) and no mute/unmuted scanner; `guardrails/pre-push` wires gate j for broad-kill but no mute-scan gate; `guardrails/README.md` does not list one; and `ROADMAP.md` carries no build-row for it (rows 327/330/334/335/333 cover the harness, the by-deed orphan helper, and the broad-kill net — none is this scan). This is a spec-vs-code divergence [INV-128, INV-144]: the spec defines the correct state, and the product is missing the net it promises. It also fails the class parity INV-160 now asserts — INV-157 must NAME a real net, and its sibling INV-162 pins its net by path (`guardrails/check-broad-kill.sh`) while this one names none. And base rule 30 (SPEC INV-164) says a machine-verifiable quality is enforced by a gate, held by no attention — the quality is named but held by nothing.

Fold: build the guardrail this landing (a tracked-script scanner, wired into `pre-push` as a new gate and into CI, the same shape as `check-broad-kill.sh`) and cite it by path in INV-157, exactly as INV-162 cites its own. If the build cannot ride this landing, soften the clause to a `[target]` and open a ROADMAP row so the spec does not claim a net that is not there. Present-tense prose plus base rule 30 argues for building it now.

`defect · behaviour-with-no-owning-node (composition)`

### F2 — The net is specified as a per-line grep, but the property it checks is not line-local, and it would red the pack's own template

> "a guardrail scans every tracked script and reds **a line** that launches a real browser without the mute flag" — INV-157, PRODUCT_SPEC.md:656

A real launch and its mute flag do not share a line. In the pack's own canonical template the launch verb is `self.proc = subprocess.Popen(` at `templates/headless_harness.py:556`, and `--mute-audio` sits inside the multi-line argv list at `:558`. A line-scoped grep of the shape INV-162's `check-broad-kill.sh` uses — match a launch line, check the same line for the flag — would flag line 556 (a launch, no `--mute-audio` on it) and **red the very template the net exists to protect**. INV-162's net is genuinely line-local (a kill verb and a bare browser name do co-occur on one line); this one is not, so the borrowed "reds a line" phrasing imports a granularity the check cannot have. A consuming maintainer wiring the gate as described would get a false red on a correctly-muted harness and either disable the gate or churn on it.

Fold: restate the net at the granularity it needs — assert that every script launching a real browser carries the mute flag in that launch's argv/config — and, as INV-162 does, leave the detection mechanism to the guardrail rather than pinning "a line". Drop "a line that launches ... without the mute flag" in favour of "a browser-launching script whose launch carries no mute flag." This also removes the false-positive on the pack's template.

`defect · over-precise-mechanism (abstraction)`

### F3 — INV-160's enumeration over-claims the net's reach against INV-157's own honest scope

> "the guardrail that scans every tracked script and reds a browser launch missing the mute flag, **catching the hand-rolled harness that diverges from the template**" — INV-160, PRODUCT_SPEC.md:660

The trailing clause reads as if the net catches a divergent harness in general. It catches only the SOUND property (the missing mute flag). A fork that mutes but leaks orphans, or sets a blanket timeout, sails past net (c) — net (a) sees only the pack template, and net (b)'s by-deed orphan check ships with the template a fork replaced, so a fork that dropped it is unguarded there. INV-157's prose home scopes this correctly ("the fork whose sound the first two nets never hear", :656); INV-160's summary loses that scope. A reader trusting INV-160 would over-estimate divergence coverage. The residual (a fork's reap/deadline divergence) is a decided answer — INV-158 says the forker owns it — so this is a wording gap, not a missing invariant.

Fold: align :660 to INV-157's scope, e.g. "catching a hand-rolled harness's **unmuted launch**" rather than "catching the hand-rolled harness that diverges from the template."

`recommendation · later · over-general (abstraction)`

## Scope honesty

Honestly stated in the INV-157 prose home (:656): the net catches the missing mute flag specifically ("the fork whose sound the first two nets never hear"), not reap or deadline divergence. No over-claim there. The single over-claim is INV-160's trailing summary clause (F3).

## "Retroactive by construction" — sound

The logic holds: a guardrail that greps the whole tracked tree on every run will, on its first run after being wired, red any pre-existing unmuted harness — the retroactive-migration property claimed. This is the proven shape of `check-broad-kill.sh`. Two conditions gate the property in practice: the guardrail must actually exist and scan the whole tracked tree (F1), and its detection must actually fire on an unmuted launch without false-firing on a muted one (F2). Given F1 and F2, the property is currently a claim about a not-yet-built, mis-specified thing; the reasoning itself is valid.
