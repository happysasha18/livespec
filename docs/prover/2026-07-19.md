### F2 — The decline-tell has no stated trigger boundary, and a broad reading re-manufactures the tidiness noise the layer exists to remove

> "when the earned-message law declines a message the agent moved to send — a candidate matching no birth, a message to a presumed-informed owner, or a question that pins to no work [INV-189, INV-190, INV-191] — the user is told that too, with the reason" — PRODUCT_SPEC.md line 1546, T-24

"A message the agent moved to send" is undefined. The disciplined agent (INV-189, INV-195) never forms a candidate that fails the births — curiosity and tidiness are suppressed before any message exists — so on the narrow reading the decline-tell almost never fires and the clause is nearly dead. On the broad reading, every suppressed "a neighbour might want to know" impulse counts as a declined candidate and earns a decline-tell with its reason. The layer's own opening (line 1438) names manufactured tidiness traffic as "the main problem this layer answers"; the broad reading moves that same traffic onto the user-facing report — a stream of "I considered telling track-coach X but withheld it because it matched no birth." That is the empty-notification noise the pack forbids elsewhere (INV-28's NEVER-list register, the whole quiet-channel intent), now on the report surface. The user reads a busy report of non-events.

State the boundary in T-24. Option a: scope the decline-tell to a candidate that reached a **named, drafted** message the agent was about to deposit (a concrete artifact existed), never a suppressed impulse — the tell then reports a real withheld file, not a thought. Option b: fold declines into the existing landing report as a single count-with-reasons line rather than a per-candidate notice, so the surface stays a summary. Option c (weakest): drop the decline-tell and let only deposits be told. My preference is (a): it keeps the visibility the clause wants (the discipline the user cannot see stays visible) while binding the tell to an observable event, so it cannot become impulse-noise. Where the line falls is the human's taste on how much of the withheld set he wants to see — SURFACED.

## Phase 3 — Property analysis (the four probed seams)

### N1 — INV-240 names the owning agent as the actor, but the re-question arrives at whatever window the human is in; the cross-window case has no route, and the one home cannot be written from a non-owning window

> "The agent whose zone owns the item reformulates that item's description so it answers the question the reader just asked, and it overwrites the description in its one home in place [base rule 4]" — PRODUCT_SPEC.md line 1454, INV-240

The trigger is "a human re-asks about a term the reference carries." A person re-asks the agent they are talking to. But the reference that confused them is very often a code owned by another window — a track-coach report cites pack code INV-28 with its description, the human asks track-coach "what is INV-28?"; INV-28's description home is the **pack's** Formal index, which track-coach cannot write (one window, one project; other repos read-only except one inbox wish file). INV-240 names the owning agent as the actor but states no transition carrying the re-question from the receiving window to that owning agent. Under base rule 4 the description has exactly one home owned by one window; under the one-window law only that window may write it. So on the most common self-heal case — a shared/neighbour code confusing an instance's reader — the named actor is absent and the receiving actor is powerless, and the description never heals. This is the exact seam the reshape opened by making the description a single-home mutable object reachable from every window's reports.

State the cross-window route in INV-240: when the re-question lands at a window that does not own the item, that window carries it to the owning agent as an **earned message** (the reader's confusion is a lived fault in the item's description, evidence in hand — a first-birth signal under [INV-189]), and the owning agent reformulates and overwrites on its own next run [INV-192, INV-195]; the same-window case overwrites directly. Option b (weaker): scope INV-240 to same-window re-questions only and accept that a cross-window confusion heals only if separately reported — this guts the feature's reach, since the confusing code is usually a neighbour's. Preference: (a). Who owns the overwrite is settled by base rule 4 — always the home's window — so the missing piece is only the route to it. Needs the design call, because it adds a transition the architecture must place.

`defect · undefined-path (transitions) + unclear-owner (actors)`

### N2 — INV-240's reactive in-place overwrite of the penned spec names only the non-empty gate as its net, and states no respect for the concurrent-edit fence or the restructure-identity gate

> "it overwrites the description in its one home in place ... The rewrite obeys the same quality bar every description obeys, and the non-empty gate stays its net [base rule 30]" — PRODUCT_SPEC.md line 1454, INV-240

The description's one home is the Formal index — a section of PRODUCT_SPEC.md, the pack's **penned shared document**. INV-240 authorizes a rewrite of it "at that moment" a reader re-asks, reactively, mid-session, with only the non-empty gate named as its net. Two collisions the clause never addresses. First, the concurrent-edit fence [INV-39, INV-198]: moving a shared document requires the pen, and a reactive description overwrite fired while another lane holds the pen (or is mid-edit on the same Formal index) is an unfenced write to a penned document — two writers, undefined behaviour by base rule 4's own "two documents claiming authority... is undefined behaviour." Second, the restructure-identity gate [INV-111]: a restructure lane's merge gate compares load-bearing token identity old-vs-new; an INV-240 rewrite that lands out-of-band changes description tokens the restructure never authored, so the gate either false-blocks on an unmatched token or, waved through, masks a real change — the literal-drift class the pack has been bitten by before. The non-empty gate catches neither; it only asserts the field is non-empty.

State in INV-240 that the in-place overwrite is a write to the description's home document and takes that document's pen before it lands [INV-198] — so a reactive rewrite queues behind an open lane rather than colliding — and that an INV-240 rewrite is a named intended delta to the restructure-merge gate [INV-111], not an unmatched token. ### N4 — the deposit-time agent-channel lint's check is undecidable on the match reading and the reshape removed the only decidable one; "without its description pair" must be pinned to a presence check

> "a deposit-time lint reads every `from-<agent>` inbox file and reds when a referenced internal code appears there without its description pair ... deterministic, blocking in CI" — PRODUCT_SPEC.md line 1450, INV-239 (net); INV-150 (deterministic bar)

INV-150 requires a mechanical net be deterministic. "Without its description pair" admits two readings. **Match**: the description beside the code equals the item's canonical description — decidable and cheap when the pair's human half was a terse *name* (string-match against the registry), but the reshape makes the human half a one-sentence description that is explicitly *translated in real time* [INV-83] and *reformulated over time* [INV-240], so a byte-match would red a legitimately translated or freshly-rewritten description, and a semantic/fuzzy match is not a deterministic gate. **Presence**: some non-empty description text accompanies the code — decidable, but blind to a wrong or garbage description. The reshape silently removed the decidable match and left the phrase reading like a match check the gate cannot perform.
