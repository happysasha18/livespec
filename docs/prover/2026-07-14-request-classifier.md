## What the delta declares

1. **INV-151 — a request enters at the highest document its change reaches, and the door set is CLOSED.**
   The derivation chain (spec → architecture → matrix → code → docs, the settings ladder beside it for a
   pure value) is walked top-down: the first document whose sentences must change is the entry layer, and
   the request flows down from there through every step its door grants. The set of entry points is
   enumerated and closed — product behaviour, a technically-phrased request, a defect, docs-only, a tiny
   reversible edit, a settings value, an inbox wish, a method/skill change, a sketch, research, and a
   feedback hand-back — each with its home and its mandatory back-check. The spec-motion back-check for a
   technically-phrased request fires AT INTAKE (the tripwires decide before entry and outrank the request's
   own label, INV-16) and re-fires mid-work, so a lift to the spec happens before architecture work is
   built on an unlifted premise. A request that matches no kind becomes one plain question (INV-4), never a
   guessed route. The method change is NOT a separate meta-layer set: the pack's product is the method, so
   a method change enters by the same criterion with work-kind skill scaling each step's form (section 8).

2. **INV-152 — a deferral must justify itself, or the item is the seat's to do.** A held work item carrying
   a needs-the-human's-word marker (a queue row, a NEXT_STEPS line, an open setup-script decision) is
   re-tested by derivability every time it is touched. An answer pinning to an existing artifact makes the
   item the seat's (do it, cite the artifact, drop the marker); an answer needing a fact no artifact holds
   (a taste, a policy, an act irreversible outside git) makes it the human's. Writing such a marker requires
   naming that human-only fact; a marker that cannot name it defaults to the seat's, and a marker that
   cannot say why the item is the human's is itself the finding — the same shape as a request matching no
   kind. Its normative home is base rulebook rule 29; the spec clause states it.

3. **INV-153 — the unification, stated once.** The request classifier (INV-151), the property net (INV-150),
   and the deferral test (INV-152) are one principle: every incoming thing routes to the home whose declared
   sentence governs it, and a thing that pins to no home is itself the finding. They stay three separate
   controls under the one principle because they run at different moments under different verifiers — the
   classifier by the applied-or-stood-down-by-name landing contract (INV-22), the property net by the
   declared-laws station (INV-101), the deferral test by the derive-before-defer posture (INV-143). No new
   machinery beyond writing the door set as closed and the fallback question.

## Findings

**F1 — the closed set lives in two homes (the spec clause and the build-pipeline table), by design (acknowledged).**
INV-151's enumeration is stated compactly in the spec clause and in full table form at build-pipeline's
door step. This is the spec-states-the-contract / skill-states-the-machinery split the pack uses everywhere
(INV-22, E-12): the table's per-kind home is build-pipeline, the spec names the closed-set law. Not a
one-home violation — the spec does not re-table the machinery, it points at the door step. `acknowledged ·
composition`

**F2 — no live inbox item exercises the deferral clause's seat-default path (acknowledged).**
INV-152 is testable in prose and its shape is exercised by the classifier's fallback, but the pack has no
live held work item today whose marker fails to name a human-only fact. The clause is exercised the day a
resume file or setup script carries such a marker; the base rule and the spec clause hold the discipline
until then. Noted so a later reader does not read the absence as a gap. `acknowledged · state-space`
