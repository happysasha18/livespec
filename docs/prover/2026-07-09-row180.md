## Findings

F1 — The architecture lens never asks for the mandated parts, so they get skipped

> "That lens checks three things: every spec fact has an owning node; no node stands without spec
> backing; every seam between nodes is named." — From the spec to the tests

An author derives an architecture doc, the prover passes it with the three-item lens, the doc ships
with no budgets, no runtime view, no placement view — exactly what happened on the tlvphoto
derivation of 2026-07-09. The reader then cannot answer "where does this run" or "how does this
flow work", and the miss surfaces only at the human's eye.

Extend the lens list in the spec's architecture-step prose, in product-prover's Phase 0
architecture-lens bullet, and in build-pipeline step 4 to six items: fact ownership · node backing ·
named seams · budgets + instrumentation home [INV-41] · runtime view per promised flow [INV-74] ·
placement per node [INV-75], each judged at the project's kind scale.

`must-fix · missing-outcome-check (postcondition)` — **folded**: spec prose + product-prover Phase 0
+ build-pipeline step 4 all carry the six-item lens.

F3 — "Every flow the spec promises" is undefined per kind

> "Both views scale by the project's kind [INV-36]." — From the spec to the tests

walks its visitor scenarios; a CLI walks one invocation per command surface; a skill pack walks a
wish through the skills; a book says its one sentence.

`should-clarify · over-general (abstraction)` — **folded**: the template's runtime-view section
names the per-kind flow unit beside the existing kind table.
