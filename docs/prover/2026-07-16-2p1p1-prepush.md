## The one named open point — judged

**Verdict: NOT sound as written. The clause needs a sentence naming the actor — chiefly for the close half.** The echo half alone would be defensible as an implementation detail (the sweeping session posts a comment under its own auth); the close half is not, and the shared "rides the monitor's channel" cite is wrong for both. Reasoning, three parts:

1. **The close has no actor and no trigger — an undischarged promise.** INV-147 now states "the Issue is closed once its row reaches a terminal exit [INV-59]." Terminal exit (landed / declined / deferred / superseded [INV-1]) happens in a later session, in build-pipeline's landing, often long after harvest. Nothing in the system connects "a row reached terminal exit" back to "close its source Issue": the monitor (`scripts/stranger-wish-monitor.py`) surfaces new items on a schedule and tracks no landed rows — I read the script, it holds no echo, no close, no row-state read; and no operating prose carries the duty — `inbox/README.md`'s sweep section (lines 17–21) names the harvest commit and says nothing of an echo or a close. So the close is a promise with no assigned actor and no home a landing session would read, the discharge/liveness class: the likeliest real outcome is a source Issue left open forever because the session that landed the row never knew it owed the close.

2. **"Rides the monitor's channel [INV-146, INV-148]" cites the wrong actor.** The clause says the writes ride "the Issue and Discussion channel the monitor already holds." Per INV-148 that channel is the scheduled Action's `github-actions[bot]` token (issues/discussions: write). But the spec's own subject is **"the sweep posts"** — the sweep is the assigned session harvesting the inbox file [T-10, INV-1], a different actor from the scheduled monitor, using its own package-repo gh auth [INV-82], not the Action token. On the package repo the split bites: the Action surfaces the bridged file but cannot harvest it (harvest is a git-atomic session act), so the Action cannot be the echo's actor either — the echo depends on a maintainer session running the sweep, which the "rides the monitor's channel" phrasing hides rather than states.

3. **Why the actor is not merely an implementation detail here.** For a purely local session behaviour the actor is implicit and rightly omitted. This clause is different on two counts: it spans two moments (harvest and a later terminal exit) with two different sessions, and it explicitly asserts a channel (INV-148's token) that neither session uses. Naming the actor is what tells a reader whether the echo is guaranteed on a repo whose surfacing is automated but whose harvest is manual, and what closes the Issue when the row lands.

Fold (the must-fix): replace the "rides the monitor's channel" clause with one that names the actor for each write and homes the close's trigger — for example: "The sweeping session posts the echo at harvest, and the session that lands the row closes the Issue at its terminal exit, each under the session's own package-repo gh auth [INV-82] — not the scheduled monitor's token, which surfaces new items and tracks no landed rows." If the close's mechanism is genuinely deferred, mark that half `[target]` so a reader of the shipped-node clause does not read an unbuilt, unassigned obligation as fact. Either resolution also wants the close duty homed where a landing session reads it (a line in `inbox/README.md`'s sweep section or the landing step), or it will be silently skipped (R1).
