## Group 2 — "a living, self-updating registry surfaced to the human"

### The proposed group

Role sentence: *a per-item store the human sees, that stays current on its own from a human signal, keeps one home, and renders its current set to the human on demand — never drifting from the truth.*

| Member | Where | Status |
|--------|-------|--------|
| the living code description | INV-240 (1454) + its field home E-35/INV-239 (1450) | proven spec |
| the living parameters registry | ROADMAP.md row 427 | OPEN DESIGN 2026-07-19, capture-only, owner-held (folds into the onboarding movement's settings window, ROADMAP 414) |

### Are they one kind? — parity of the four lifecycle behaviours

| Behaviour | living code description [INV-240] | living parameters registry [ROADMAP 427] |
|-----------|-----------------------------------|------------------------------------------|
| how an entry is **born** | at code registration, authored into the Formal-index description field | when an incoming request is recognized as adjusting a parameter |
| how it is **updated** | on a human re-question ("what is this?") — the signal the description did not land | on an incoming request that adjusts the parameter's value |
| how it is **shown** | inline beside the code in every reference [E-35] | in a live settings window, rendered on demand |
| how it is **overwritten** | in place, in its one home (the description field) [base rule 4] | the one live registry stays current, in place |
| its **net** | non-empty gate reds an empty description [base rule 30] | (undesigned — presumably a no-drift / non-empty gate) |

Structurally the shape is identical: **one home, an entry born from a declared signal, updated in place from that same class of signal, rendered to the human on demand, and it must never drift.** The differences are all *fillings* of that shape (born-from-authoring vs born-from-traffic; shown-inline vs shown-in-a-window; update-on-re-question vs update-on-request), not differences in the shape itself. The text nowhere says these two are one kind, so 427 — still unbuilt — is free to grow born/updated/shown/overwritten/net answers that diverge from INV-240's for no reason. See ASK-2. Note 427 is a ROADMAP capture, not a proven clause, so this ask is forward-looking: its natural home is the owner's design of 427 inside the onboarding movement, and it rests `held` until then.

## Findings on the echo channel (strongest first, at most three)

### ASK-1 — where do the deposit-tell and decline-tell live? *(likely — question to the human)*

- **Object A — the deposit-tell.** "An automatic deposit is a step the user did not watch, so the sending agent tells its own user that a message reached the neighbour's inbox ... a plain notice in the report register, never a confirmation of a deposit already made [INV-28, INV-31]." (PRODUCT_SPEC.md:1550, T-24) — **home unnamed; cites INV-31 = the landing-report law.**
- **Object B — the escalation surfacing.** "An escalated message surfaces in the sender's own status report as blocked work aged past its stated need-by ... The human reads that line." (PRODUCT_SPEC.md:1577, INV-192 / INV-27) — **home: the status report, explicit.**
- **Shared role:** a terminal or lifecycle event of an agent-channel message [INV-183] the user did not watch, surfaced to the user after the fact.
- **Divergence:** Object B names the status report as its home; Object A (and its twin the decline-tell) names no report home at all, and the only home it cites is INV-31, the *landing*-report taste-told law — a home built for a different sub-kind (a retunable choice), pulling the placement away from where its own agent-channel siblings (escalation, wrong-referral) live.
- **How alike should these behave?** Should the deposit-tell and decline-tell home in the **status report**, beside the escalation and the wrong-referral (their agent-channel message-lifecycle siblings [INV-183, INV-27]), or in the **landing report** with the taste-told family [INV-31]?
- **Recommended default:** the **status report**. A deposit and a decline are message-channel events on the agent channel [INV-183], on the sender's own status cadence rather than tied to a build landing — the same footing as the escalation and the wrong-referral that already home in the status report [INV-27]. T-24's cite of INV-31 should be read as borrowing only the *notice-not-confirmation register*, not the placement. If the human instead reads a deposit as a landing-time event, the landing report is the alternative — but then the escalation/deposit split needs a stated reason.
- **Bar check:** one plain shared-role sentence ✓; a whole-behaviour difference (a named report home vs none) ✓; no spec sentence decides it — T-24 leaves the home unnamed ✓.
- **Outcome:** `asked`

### ASK-2 — is the living parameters registry (ROADMAP 427) a member of the living-code-description's kind? *(likely — question to the human/owner)*

- **Object A — the living code description.** "When a human re-asks about a term the reference carries ... the agent whose zone owns the item reformulates that item's description ... and overwrites the description in its one home in place [base rule 4] ... netted by a gate that reds a registered code with no description [base rule 30]." (PRODUCT_SPEC.md:1454, INV-240; field home E-35/INV-239)
- **Object B — the living parameters registry.** "The pack keeps one live registry of every per-person parameter a host may tune, and an incoming request is recognized for whether it adjusts a parameter, updating the registry the moment it does, so the list never drifts from the truth and the settings window renders the current set on demand." (ROADMAP.md:315, row 427 — OPEN DESIGN, capture-only)
- **Shared role:** a per-item live registry the human sees, kept current on its own from a human signal, one home, rendered to the human on demand, never drifting.
- **Divergence:** the two carry the same registry shape but the text never declares them one kind, so 427's still-open design of *how an entry is born, updated, shown, overwritten, and netted* could diverge from INV-240's for no reason.
- **How alike should these behave?** Should ROADMAP 427 be designed as a **member of the same living-registry class** as INV-240 — same born-from-a-declared-signal, updated-in-place, rendered-on-demand, one-home, must-not-drift behaviour, carrying a net that reds a stale or empty entry [base rule 30]?
- **Recommended default:** yes — design 427 as a living-registry sibling of INV-240 so the two share the born/updated/shown/overwritten/net shape by construction, and 427 inherits a no-drift/non-empty net rather than inventing one. The fillings legitimately differ (born from recognized traffic, shown in a settings window); the *shape* should be one.
- **Bar check:** one plain shared-role sentence ✓; a whole-shape kinship, not a parameter tweak ✓; no spec sentence decides it — 427 is capture-only and unbuilt ✓. Caveat: 427 is a ROADMAP capture, not a proven clause, so this is forward-looking design guidance for its owner rather than a divergence inside the proven spec.
- **Outcome:** `held` (forward-looking; resumes when the owner designs 427 inside the onboarding movement)

## Confident recommendation (queues for a taste call, does not block)

### REC-1 — add a class clause naming "the after-the-fact user-tell" family and its home-keying rule

The eleven members in Group 1 share one role and one register but their report **home** is decided ad hoc per member and left unnamed for the two newest (deposit-tell, decline-tell). Recommend spec-author add a class-level clause (the form the uniformity check demands [INV-125]) that:

1. Names the class — *every agent action the user did not watch, surfaced to the user after the fact* — and enumerates its members (taste choice [INV-31], tunable parameter [INV-70], economy shed [T-19], footprint re-classification, substance removal [INV-109], step stand-down [INV-22], plan-vs-prototype diff [INV-30], escalation [INV-192], wrong referral [INV-225], deposit-tell, decline-tell [T-24]).
2. States the two shared properties already true of all of them: the plain report register with no bookkeeping numbers as content, and the user named by role, never a person [INV-28].
3. States the one property the members do **not** yet share explicitly — the report **home**, keyed by cadence: a landing-time choice or record homes in the **landing report** [INV-31]; a status-cadence agent-channel message-lifecycle event homes in the **status report** [INV-27]. Placing the deposit-tell and decline-tell under this rule resolves ASK-1 (default: status report) mechanically for any future sibling.

This is a recommendation, not a blocker — it queues for the human's taste call, and the class sentence lands only on the human's word, after which the prover's uniformity check [INV-125] holds the home-keying rule for every future tell.
