## The one question (finding 1), both objects in hand

`scripts/sync-mirrors.sh` writes **three** generated blocks onto each standalone mirror every sync: the read-only-mirror **banner** (README line 1), the **release-history** section [INV-181], and the **attribution** line [INV-96]. All three are generated fresh, wiped by the `rsync --delete` rebuild from the pack's clean copy, and re-stamped each run. Two of the three — release-history and attribution — are declared spec clauses, each with a dedicated test. The **banner** has neither a spec clause nor a test, and it alone is not version-derived (its wording is keyed to the skill name, not to VERSION or git history).

**Are these three one declared kind — should the banner join as a declared, tested third member of the sync-generated mirror content family — or is the banner deliberately left as script-only mirror hygiene (a navigation notice, not a published claim like who-made-this and what-shipped-when), and so kept out of the spec on purpose?**

Recommended default: **declare and test the banner as a lighter third member.** Ground: an undeclared, untested generated block on a public surface drifts silently — a wording change or a bug in `banner_for` has no net, where its two siblings each red on drift; and INV-181 already treats the family as a unit by cross-linking INV-96. Keep the clause lighter than the pair (it carries no version/claim, so it owes no adoption-tracker or per-version duty) — one sentence that the banner is generated fresh each sync and a test that it lands on README line 1. This also settles the kind's definition: if the human reads the kind as "version-derived generated mirror content," the banner is deliberately out and the answer is "different, by intent — script-only hygiene," recorded as a decided sentence so the ask never re-fires. Either answer lands a declaration; the class sentence lands only on the human's word [INV-142], and the lane is not held.
