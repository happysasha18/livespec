# What live-spec can carefully borrow — survey of mature spec/agent-driven methods (2026-07-05)

Research question: which pieces of BMAD, spec-kit, Kiro, the OpenAI/Anthropic agent guides, Shape Up,
and the Google document tradition are worth borrowing into live-spec's pipeline
(wish → spec-delta + facet sweep → prove → architecture → prove → matrix → test → code → verify → pre-push gates)?

Method: primary sources only — the actual repos, templates, and official docs, fetched 2026-07-05.
Complements `docs/prior-art.md` (which asked "is our claim novel?"); this asks the opposite:
"what do THEY do better?". Honest-overlap notes mark what we already have so nobody re-invents it.

---

## 1. GitHub spec-kit

What it is: GitHub's open-source spec-driven-development toolkit — a CLI plus templates driving
`/constitution → /specify → /clarify → /plan → /tasks → /analyze → /implement`, each phase producing a
validated artifact before the next runs. Source: https://github.com/github/spec-kit and
https://raw.githubusercontent.com/github/spec-kit/main/templates/spec-template.md

Borrowable ideas:

- **A named `clarify` step with coverage-based questions.** Before planning, a structured questioning
  pass sweeps the spec for ambiguity by category and records answers in a Clarifications section —
  "required before planning to reduce rework downstream." We have ask-never-guess and batched questions,
  but not a question *taxonomy* that guarantees coverage; our questions come from whatever the author
  noticed. (spec-kit README, workflow section)
- **`[NEEDS CLARIFICATION]` markers inside the spec text.** Ambiguity is flagged in-place, mechanically
  findable, instead of living in the author's head. Our `[default]` tag is the sibling for *answered*
  holes; we have no marker for *open* ones — an open question currently blocks the lane instead of being
  parked visibly. (spec-template.md)
- **Measurable Success Criteria (SC-001…) as a mandatory spec section.** Technology-agnostic, measurable
  outcomes separate from functional requirements — "how would we notice this feature worked for the
  person." Our SPEC has scenarios and invariants but no per-feature success measure. (spec-template.md)
- **`/analyze`: a cross-artifact consistency check run before implementation.** Verifies spec ↔ plan ↔
  tasks agree. Our pre-push guardrails check matrix coverage and prover records; a spec ↔ ARCHITECTURE ↔
  TEST_MATRIX cross-consistency pass exists only as prover convention, not as a script. (spec-kit README)
- **Prioritized, independently-testable user stories (P1/P2/P3).** "Each user story/journey must be
  INDEPENDENTLY TESTABLE" — a standalone slice, deployable and demonstrable alone. This is Alexander's
  one-feature-one-story rule stated as a template constraint; the priority ladder on top is the new bit.
  (spec-template.md)

Already have: spec-before-code, per-requirement anchors, feature-scoped spec deltas, a constitution
equivalent (live-spec-base + profile ladder), TDD ordering in tasks.

## 2. BMAD-METHOD

What it is: "Breakthrough Method for Agile AI-Driven Development" — a pack of 12+ role agents
(analyst, PM, architect, PO, scrum master, dev, QA) with document templates and *named checklists* that
gate each hand-off. Source: https://github.com/bmad-code-org/BMAD-METHOD; checklists (V4 branch):
pm-checklist, architect-checklist, po-master-checklist, story-dod-checklist, qa-gate template.

Borrowable ideas:

- **The QA gate verdict ladder: `PASS / CONCERNS / FAIL / WAIVED`.** A gate decision is a YAML record
  with `top_issues` (id, severity, finding, suggested_action) and a `waiver` field carrying `reason` +
  `approved_by`. Our prover has must-fix/should-clarify severities but a review ends binary — there is no
  first-class "owner saw the risk and accepted it" record, so accepted risks either block or evaporate.
  (bmad-core/templates/qa-gate-tmpl.yaml, V4)
- **NFR checklist dimensions we don't sweep.** The pm-checklist's Non-Functional Requirements section
  and the architect-checklist force: security & compliance, reliability & resilience, monitoring/
  observability/alerting, data entities & relationships, external integrations, deployment needs. Our
  five-facet sweep covers layout/touch/states/a11y/perf — candidates from BMAD are listed in the facet
  table below. (bmad-core/checklists/pm-checklist.md, architect-checklist.md, V4)
- **Self-contained story files for the implementing agent.** The scrum master compiles each story into
  one .md carrying the full context the developer needs — the relevant PRD and architecture excerpts
  embedded, not referenced. Directly applicable to our sonnet-worker delegation: a worker brief that
  embeds the spec sentences + matrix rows it implements would remove the "worker lacks context" excuse.
  (BMAD user guide / core workflow)
- **A definition-of-done the implementer must *honestly self-assess*.** story-dod-checklist ends with the
  agent summarizing what's genuinely done, listing incomplete items with reasons, flagging tech debt —
  before requesting review. Our verify step drives the app but has no structured self-confession format.
  (bmad-core/checklists/story-dod-checklist.md, V4)
- **`[[BROWNFIELD ONLY]]` / `[[GREENFIELD ONLY]]` conditional checklist items.** The PO master checklist
  adapts by project type instead of maintaining two lists. Our adopt mode vs greenfield could mark
  sweep/pipeline steps the same way. (bmad-core/checklists/po-master-checklist.md, V4)

Already have: the PO-style master validation (our prover), role separation (spec-author / prover /
pipeline / communicator), sequencing checks (matrix coverage gate).

## 3. AWS Kiro (spec mode)

What it is: AWS's agentic IDE whose spec mode drives every feature through three living files —
requirements.md (user stories + EARS acceptance criteria), design.md (architecture, sequence diagrams,
error handling, testing strategy), tasks.md (dependency-ordered tasks tracing back to requirements) —
with approval gates between phases. Source: https://kiro.dev/docs/specs/ and
https://kiro.dev/docs/specs/best-practices/

Borrowable ideas:

- **EARS-style regression clauses: "WHEN [condition] THEN the system SHALL CONTINUE TO [existing
  behavior]".** For bugfix work, Kiro makes you write down what must NOT change before touching code.
  This is our reverse-verify pain (spec↔shipped drift) turned into a spec sentence: every wish that
  touches an existing surface gets explicit continue-to clauses, and the matrix derives regression rows
  from them. (kiro.dev/docs/specs/best-practices)
- **Structured acceptance-criteria notation (EARS: WHEN/IF…THEN…SHALL).** One grammatical shape for
  every testable claim makes machine-derivation of test rows reliable. Our anchors identify claims; EARS
  would normalize their *shape*. Borrow carefully — plain-language-first is our rule; EARS can live in
  the Formal index only. (kiro.dev/docs/specs)
- **Task waves from a dependency graph.** Kiro groups independent tasks into concurrent "waves" —
  a ready-made pattern for fanning matrix rows out to parallel cheap workers. (kiro.dev/docs/specs)
- **Design.md must state error handling and testing strategy as sections, not afterthoughts.** Our
  ARCHITECTURE.md has no mandatory error-handling section. (kiro.dev/docs/specs/concepts)

Already have: the three-artifact split (SPEC / ARCHITECTURE / TEST_MATRIX), approval gates between
phases (prover checkpoints), continuous refinement of specs as living documents, task↔requirement
traceability (anchors + coverage gate).

## 4. Anthropic — agent engineering guides

What it is: Anthropic's "Building effective agents" (patterns, when to add complexity) and the Claude
Code best-practices doc (the operating manual for agentic coding sessions). Sources:
https://www.anthropic.com/engineering/building-effective-agents and
https://code.claude.com/docs/en/best-practices

Borrowable ideas:

- **"Give Claude a check it can run" — a graded gate ladder.** Same check, four enforcement levels:
  in-prompt → per-turn goal condition → deterministic Stop hook that blocks until the script passes →
  independent reviewer in fresh context. We enforce at pre-push only; a Stop-hook-grade check at the
  *end of each pipeline stage* (not just before push) is the missing middle rung. (best-practices,
  "Give Claude a way to verify its work")
- **Anti-over-engineering instruction to reviewers.** "A reviewer prompted to find gaps will usually
  report some, even when the work is sound… Tell the reviewer to flag only gaps that affect correctness
  or the stated requirements, and treat the rest as optional." Our prover severity rule (operational
  impact, not formal imperfection) is the same spirit — the explicit "report gaps, not style
  preferences" phrasing and the warning about chasing findings into defensive code are worth adopting
  verbatim into product-prover. (best-practices, adversarial review callout)
- **The interview pattern for spec intake.** "Interview me in detail… don't ask obvious questions, dig
  into the hard parts I might not have considered" — then write the spec and execute in a *fresh
  session*. A ready shape for wish intake on bigger wishes. (best-practices, "Let Claude interview you")
- **Evidence over assertion.** "Have Claude show evidence rather than asserting success: the test
  output, the command it ran, a screenshot." Matches our show-real-not-synthetic rule; the delta is
  making evidence a *required field* of every stage report. (best-practices)
- **Complexity ratchet.** "Start with simple prompts… add multi-step agentic systems only when simpler
  solutions fall short" — the justification for our tiny-reversible-edit shortcut lane; keep it guarded.
  (building-effective-agents)

Already have: fresh-context adversarial review (prover), subagent fan-out, deterministic pre-push
hooks, checkpoint discipline.

## 5. OpenAI — "A practical guide to building agents"

What it is: OpenAI's official PDF guide on designing production agents — orchestration patterns, a
guardrails taxonomy, and human-in-the-loop policy. Source:
https://cdn.openai.com/business-guides-and-resources/a-practical-guide-to-building-agents.pdf

Borrowable ideas:

- **Explicit human-intervention triggers: failure thresholds and high-risk actions.** Escalate to a
  human when (a) retries/failures exceed a threshold, and (b) the action is irreversible or high-stakes.
  For live-spec: classify pipeline actions by *reversibility* — irreversible ones (push, publish, delete,
  spend) always stop for the owner; reversible ones ride the proactivity dial. This gives the
  MAX-PROACTIVE mode a principled boundary instead of a vibe. (guide, safety section)
- **Layered guardrails taxonomy.** Relevance classifier, safety classifier, PII filter, rules-based
  protections (blocklists, input limits), tool-risk rating, output validation — defense in depth, each
  cheap check running before the expensive one. Our guardrails/ scripts are one layer; the idea of
  *rating each tool/step by risk* and gating proportionally is the borrowable structure. (guide,
  guardrails section)
- **Single-agent first, multi-agent only on demonstrated need** — same ratchet as Anthropic; supports
  keeping the pack four skills, not twelve BMAD-style roles. (guide, orchestration section)

## 6. Shape Up (Basecamp)

What it is: Basecamp's product method — shaped pitches bet on in six-week cycles. The pitch has five
ingredients: problem, appetite, solution, rabbit holes, no-gos. Source: https://basecamp.com/shapeup
(ch. "Write the Pitch")

Borrowable ideas:

- **Appetite: fixed time, variable scope, declared up front.** "If we only care enough to spend two
  weeks on this now, how does this specific solution look?" A wish that declares its appetite lets every
  downstream default be sized honestly — the facet sweep's `[default]` choices should differ for a
  half-day wish vs a two-week one. Nothing in our intake records how much the owner cares. (ch. 06)
- **No-gos: what the feature deliberately will NOT do, written in the spec.** Google's non-goals twin.
  Our SPEC states what IS; nothing forces "reasonable expectations we explicitly refuse" onto paper, so
  the prover can't tell a hole from a refusal. (ch. 06)
- **Rabbit holes: known risks flagged before work starts.** A named section for "this part could blow
  up" — cheap insurance we currently leave implicit in prover findings *after* drafting. (ch. 06)
- **Fat-marker sketches / breadboards: deliberately low fidelity when asking for decisions.** "Stay high
  level, but add a little more concreteness" — so the owner decides the idea, not the pixel. Our
  communicator already shows real-data mockups; the calibrated-fidelity idea (rough on purpose while the
  decision is conceptual) is the addition. (ch. 05–06)

## 7. Google engineering document tradition

What it is: three primary artifacts — the design-doc anatomy (context, goals/non-goals, design with
trade-offs, alternatives considered, cross-cutting concerns), the code-review standard, and the SRE
launch checklist. Sources: https://www.industrialempathy.com/posts/design-docs-at-google/ ·
https://google.github.io/eng-practices/review/reviewer/looking-for.html ·
https://sre.google/sre-book/reliable-product-launches/

Borrowable ideas:

- **Goals AND non-goals as paired mandatory sections.** Non-goals are "things that could reasonably be
  goals but are explicitly not" — the strongest single guard against scope drift and prover
  false-positives. (design-docs post)
- **Alternatives considered, with why the winner won.** The design doc is "the place to write down the
  trade-offs"; alternatives get a paragraph each. Our docs/decisions/*.json record choices; the
  discipline of recording the *losing* options and the reason is the delta. (design-docs post)
- **Mandated cross-cutting concerns: security, privacy, observability.** Short mandatory sections; at
  Google privacy gets a separate document and a dedicated review before launch. Three facet candidates
  with pedigree. (design-docs post)
- **The launch-checklist curation rule: every item justified "ideally by a previous launch disaster",
  regularly pruned.** Governance for our canonical facet list — a facet earns its slot by a named
  real incident and is re-justified periodically, which is exactly how the list stays five-ish long
  instead of becoming a 40-row form. (SRE book ch. 27)
- **Launch checklist dimensions:** architecture & dependencies, capacity, *failure modes*, *client
  behavior*, external dependencies, *rollout/rollback planning* — the failure-modes and
  external-dependency questions ("what happens when the thing you rely on is missing/slow/wrong?") are
  absent from our sweep. (SRE book ch. 27)
- **Code-review dimensions include edge cases "looking for concurrency problems" and every-line
  coverage.** Concurrency-of-instances (two windows, same stored state) is a proven bug class for us —
  the persisted-aim bug in track-coach was exactly stored-state × a second surface. (eng-practices)

Already have: trailing-anchor traceability, decision records (thinner than Google's), the stored-state
version-N-1 migration axis in spec-author (a genuine overlap with "failure modes" — we cover *older
data*, not *missing dependencies*).

## 8. Amazon Working Backwards (PR/FAQ)

What it is: Amazon's mechanism since ~2004 — before building, write a one-page future-dated press
release in plain customer language plus an FAQ that holds the hard questions (problem, risks, success
measures). Source: https://workingbackwards.com/resources/working-backwards-pr-faq/ (Bryar & Carr,
the format's authors).

Borrowable ideas:

- **The FAQ as the tradeoff channel to a non-technical owner.** Hard decisions are surfaced as questions
  a skeptical reader would ask, answered in customer terms — not as a settings list. Our defaulted-facet
  report ("on a phone this gallery stacks into one column — ok?") is already this pattern; the FAQ frame
  generalizes it to *all* tradeoffs, not just facet defaults. (workingbackwards.com)
- **"No jargon, no internal acronyms" as a hard rule for the owner-facing page** — independent
  confirmation of our plain-language + trailing-anchors choice; their test is "would a customer
  understand every sentence?". (workingbackwards.com)
- **A hypothetical customer quote in the spec's opening.** One sentence of the person using the shipped
  feature — a cheap concreteness test: if the quote can't be written, the feature isn't understood.
  (PR/FAQ template)

Already have: use-case-first scenarios leading the spec (our whole spec shape IS working-backwards).

## Candidate facets others carry that our five-facet sweep lacks

Current canonical list (spec-author SKILL.md): phone/narrow layout · touch-vs-hover ·
empty/error/loading · accessibility · performance envelope. Plus, outside the sweep, the stored-state
version-N-1 axis. Candidates, each one line + source:

| Candidate facet | Why | Source |
|---|---|---|
| **Concurrent instances** — two windows/tabs/sessions on the same stored state | our own persisted-aim bug class; "edge cases, looking for concurrency problems" | Google eng-practices; live-spec fence rules already gate concurrent *edits*, not concurrent *runtime* |
| **Dependency failure** — the file/tool/service the feature reads is missing, renamed, or stale | "failure modes" + "external dependencies" are whole launch-checklist areas; matches "ask where a missing source is, don't guess" | SRE book ch. 27 |
| **Security / untrusted input** — what happens when input is hostile or malformed, who may do this | mandated cross-cutting concern; NFR checklist staple | Google design docs; BMAD pm-checklist §5 |
| **Privacy / data retention** — what personal data the feature stores, for how long, who can see it | the one concern Google mandates as a *separate reviewed document* | Google design docs |
| **Observability** — how we'd notice in the wild that the feature broke | third mandated Google concern; BMAD "monitoring, observability, alerting"; for local tools = "what does the user see when it silently fails?" | Google design docs; BMAD architect-checklist §5 |
| **Undo / destructive-action recovery** — can the person reverse it; irreversible ⇒ confirm | "error handling and recovery approaches" (BMAD UX); irreversible-action escalation (OpenAI) | BMAD pm-checklist §3; OpenAI guide |
| **Rollout / rollback** — how the change lands and how it comes back out | dedicated launch-checklist area; we do this ad hoc (attic copies) | SRE book ch. 27 |
| **Success measure** — how we'd notice the feature worked for the person | mandatory Success Criteria section (SC-001…) | spec-kit spec-template |
| **Discoverability / first-run** — how the person finds the feature the first time | user-flow + documentation items in PM checklist and story DoD | BMAD pm-checklist §3, story-dod §7 |
| **i18n / text expansion** | classic launch dimension; probably N/A for this project family — listed for completeness, not recommended now | launch-checklist tradition |

Governance note: do NOT add all of these. Apply the SRE curation rule — a facet joins the canonical
list only with a named incident (concurrent-instances and dependency-failure both have one), and the
list is re-justified periodically. The rest can live as a "long list" the clarify step samples from.

## How they report tradeoffs to a non-technical owner — patterns worth naming

- **Question-shaped, in customer language** (Amazon FAQ): raise the tradeoff as the question a skeptical
  customer would ask, answer in their words. We already do this for facet defaults; extend to all defaults.
- **Appetite-anchored** (Shape Up): "for the two weeks we're willing to spend, this is the shape" — price
  and scope in the same sentence, so the owner vetoes the *bet*, not the implementation.
- **Non-goals stated as sentences** (Google/Shape Up): "it will not X" written down, so silence is
  distinguishable from refusal.
- **Waiver records** (BMAD): accepted risk gets `reason` + `approved_by` — the owner's "yes, ship anyway"
  is a durable artifact, not a chat message.
- **Calibrated fidelity** (Shape Up fat-marker): show rough when the decision is conceptual, real-data
  when the decision is visual — matches communicator's show-real rule, adds the "rough on purpose" half.

## Top 10 candidates for live-spec (ranked)

1. **Regression fence sentences: every wish touching an existing surface writes "SHALL CONTINUE TO …"
   clauses before work starts.** Why: turns our reverse-verify pain (spec↔shipped drift, the recurring
   Simple/Detailed break) into spec text the matrix mechanically derives rows from.
   Source: Kiro spec best practices (kiro.dev/docs/specs/best-practices).
2. **Prover verdict ladder PASS / CONCERNS / FAIL / WAIVED, with waivers carrying reason +
   approved-by.** Why: today a prover finding either blocks or evaporates; a waiver record makes the
   owner's accepted risk durable and auditable. Source: BMAD qa-gate-tmpl.yaml (V4).
3. **Mandatory no-gos/non-goals in every feature's spec-delta.** Why: the prover (and a future session)
   can't currently tell a hole from a deliberate refusal; one sentence fixes a whole class of
   false findings and scope drift. Source: Shape Up ch. 06; Google design docs.
4. **Appetite declared at wish intake.** Why: fixed-time-variable-scope makes every downstream default
   honest — the same facet defaults differently for a half-day wish vs a two-week one.
   Source: Shape Up ch. 06.
5. **Facet-list curation rule: a facet joins the canonical list only with a named real incident, and the
   list is periodically re-justified.** Why: keeps the sweep five-ish strong facets instead of decaying
   into a 40-row form nobody reads. Source: Google SRE book ch. 27 (launch checklist curation).
6. **Two new facets now, by that rule: concurrent instances (two windows on the same stored state) and
   dependency failure (the source the feature reads is missing/renamed).** Why: both have named
   incidents in our own projects (persisted-aim bug; missing-source guessing).
   Sources: Google eng-practices (concurrency in edge cases); SRE launch checklist (failure modes,
   external dependencies).
7. **Self-contained story briefs for delegated workers — the worker's file embeds the exact spec
   sentences + matrix rows it implements.** Why: attacks our standing sonnet-delegation failure by
   removing the context excuse and making delegation mechanical. Source: BMAD scrum-master story files.
8. **Measurable success criterion per feature (one SC sentence: how we'd notice it worked).** Why: our
   spec proves consistency but never states what "worked" means for the person; also the seed of any
   later telemetry facet. Source: spec-kit spec-template (Success Criteria, SC-001…).
9. **Reviewer anti-over-engineering clause in product-prover, verbatim spirit: "flag only gaps that
   affect correctness or the stated requirements; report gaps, not style preferences".** Why: a prover
   asked to find gaps will always find some; without this, findings-chasing breeds defensive code.
   Source: Anthropic Claude Code best practices (adversarial review callout).
10. **Irreversibility rule for proactivity: irreversible actions (push, publish, delete, spend) always
    stop for the owner regardless of the proactivity dial; reversible ones ride it.** Why: gives
    MAX-PROACTIVE a principled boundary instead of a per-case judgment call — and matches the push gates
    we already improvise. Source: OpenAI "A practical guide to building agents" (human-in-the-loop
    triggers); echoed by Anthropic's gate ladder.

Near-misses (real value, not top-10): spec-kit's `/clarify` coverage taxonomy and `[NEEDS
CLARIFICATION]` open-hole marker; Kiro's dependency-graph task waves for parallel workers; BMAD's
honest self-assessment DoD; Google's alternatives-considered discipline in decision records; Amazon's
FAQ frame for all owner-facing tradeoffs; per-stage deterministic gates (Anthropic Stop-hook rung)
between pipeline phases rather than pre-push only.

Already ours, confirmed against sources — no need to borrow: spec-before-code with living updates;
prover as fresh-context adversarial review; mechanical pre-push gates (matrix coverage, prover record,
tests, prototype fence); plain-language owner reports with `[default]`-tagged batched tradeoffs;
use-case-first spec shape (≈ working backwards); constitution equivalent (live-spec-base + profile
ladder); stored-state version-N-1 migration axis; one-feature-one-story (≈ spec-kit's independently
testable stories).
