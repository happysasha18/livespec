# Adversarial audit — batch be70e35..HEAD (pre-MINOR gate 2.1.1 → 2.2.0)

Scope: the six commits since v2.1.1 (prover-doc restructure + M-338, row 359 extractor fix, row 360 mirror
release history, the 2.2.0 first gate-folds commit) plus the uncommitted folds present at audit time
(INV-181 three-sentence growth, M-340, the release-history test's skip-guard and banner test, CHANGELOG in
CAPS_ALLOW). Method: refute-first — every claim below was attacked before it was accepted. All constructions
were run against the real helpers/scripts; raw outputs quoted. Suite state at audit: 940 passed, 0 failed.

## Defects

### D1 — the generated release history shows an unreleased version as a release (live, HIGH)
The harvest regex (`scripts/sync-mirrors.sh:162`) matches ANY subject leading with a semver, release or not.
The batch's own gate-folds commit — subject `2.2.0 gate folds, first batch: …` — is harvested, and
`./scripts/sync-mirrors.sh --print-release-history` now prints as its FIRST line:

    - 2.2.0 · 2026-07-16 — gate folds, first batch

while `VERSION` reads `2.1.1` and no 2.2.0 shipped. A sync run today would stamp a phantom 2.2.0 release
onto every public mirror. Worse, the dedupe (`:183-190`, oldest-matching-commit wins) means that when the
real 2.2.0 bump commit lands LATER, it loses to this commit — "gate folds, first batch" becomes 2.2.0's
permanent story. Violates INV-181's "one line per shipped version" (PRODUCT_SPEC.md, the INV-181 paragraph)
and M-339's NEVER arm. Fix direction: harvest only versions ≤ VERSION (or only versions with a JOURNAL
chapter / a tag), and prefer the bump commit explicitly over positional dedupe.

### D2 — the dedupe's "oldest matching commit = the bump commit" premise is false in the real log (MEDIUM)
Script comment `:86-88` claims the oldest matching commit per version is "the bump commit itself".
Refuted live: 0.9.0's oldest matcher is `697965c` "0.9.0 milestone audit: fold four craft/dedup findings…"
(a PRE-bump audit commit); the actual bump is the newer `a4c3e78` "0.9.0 milestone: MINOR bump…". The
printed line is therefore

    - 0.9.0 · 2026-07-08 — milestone audit

— a non-release commit's subject shown as the release's story, exactly what M-339's Fact column says NEVER
happens (TEST_MATRIX.md, M-339). The owning test pins only the 2.0.0 follow-up case ("prover record" absent)
and misses this one. Same mechanism as D1's future-story corruption.

### D3 — the harvest is locale-sensitive; under LC_ALL=C every em-dash subject leaks mojibake (MEDIUM)
`LC_ALL=C ./scripts/sync-mirrors.sh --print-release-history` (this machine, /bin/bash 3.2.57):

    - 2.1.1 · 2026-07-16 �M-^@M-^T M-^@M-^T the day-after sweep      (cat -v)

The bracket `[:—-]` in the `=~` expression (`:162`) matches single BYTES of the multibyte em-dash in the
C locale, splitting it and leaking the remainder into the story. The script documents a CI arm
(`MIRROR_SSH=1`, `:249-252`) where C locale is the common default, so this is a real deployment path, not a
lab case. Fix: pin the locale at the top of the script (or match the separator without a bracket class).

### D4 — the extractor's bare-name arm keeps the M-146 substring false-green; one live row rides it (MEDIUM)
Row 359 claims the truncation false-green closed at root. Only the file-path FORM closed. The bare-name arm
still uses substring matching (`tests/test_traceability.py`, `_check_owning_row_tests`:
`assertIn("def %s" % name, this_file)`), so a cited name that is a PREFIX of a real def passes. Run raw:

    A bare-name prefix of a longer def: GREEN   ← `test_matrix` vs `def test_matrix_blocks_match(): pass`

Live instance found by sweeping the real matrix: **M-301** (TEST_MATRIX.md:507) cites
`test_flaky_test_is_a_defect`; no def by that exact name exists — only
`test_flaky_test_is_a_defect_spec_states_the_seam` / `_build_pipeline_green_definition` /
`_test_author_determinism_rule` (tests/test_flaky_test_is_a_defect.py). The row is green today by the same
substring luck M-146 rode. Fix: exact-def match (`def <name>(` or a word boundary) for bare names; M-301's
cell then either names the real defs or the file path.

### D5 — M-325's Fact column describes the pre-restructure structure as current, contradicting M-338 (MEDIUM)
TEST_MATRIX.md:202 (M-325) still claims "the KIND block splits the declared one-sided pair … and lets an
open motion question ride" — but this batch moved that split OUT of the KIND block into the
Paired-transition symmetry lens, and M-338's Fact pins "NEVER the family split leaking back into KIND". Two
matrix rows now assert opposite locations for the same text. The owning test
`test_kind_block_names_motion_question_gate_semantics` (tests/test_audit_point_folds.py:32) greps the WHOLE
file, so it stayed green through the move — its name asserts a location it never checks; the file's
docstring ("R1: the KIND block names…") is stale the same way. Fix: reword M-325's Fact (and the test
name/docstring) to the lens home; the strings themselves survived (see refuted-claims below).

### D6 — ROADMAP row 360 asserts a verification that has not happened (MEDIUM)
Row 360's Done-when reads "MET (verified on the live mirror after the 2.2.0 push)". The 2.2.0 push has not
happened; the live mirror's last commit is "sync from live-spec pack 2.1.1 (179d2d7)" (2026-07-16T13:49:52Z,
`gh api repos/happysasha18/product-prover/commits`) and its README carries NO release-history section (only
the attribution line at :152). History asserted ahead of the deed — the row should read MET-pending-push or
name the local `--print-release-history` verification it actually has.

### D7 — the new git-repo skip-guard skips wrongly in a git worktree (LOW)
tests/test_mirror_release_history.py:27-29 gates the by-deed test on `os.path.isdir(ROOT/.git)`. In a
linked git worktree `.git` is a FILE, not a directory — the guard skips the test there even though the git
history is fully readable and the generator would run fine. `os.path.exists` (or `git rev-parse
--is-inside-work-tree`) is the correct precondition; as written, an isolated-worktree gate run silently
loses this test's coverage.

## Recommendations

### R1 — INV-164 posture: the phantom-release quality is machine-verifiable and no gate holds it
The suite is 940-green with D1's phantom 2.2.0 line present. One cheap standing test closes the class:
assert the generated section's first line's version equals the VERSION file (and/or every printed version
has a `## ` JOURNAL chapter). This also nets D2 (assert 0.9.0's story comes from its bump commit, or drop
the positional dedupe for an explicit bump-subject rule).

### R2 — the path regex lacks a trailing boundary
`tests/test_traceability.python notes` harvests as `tests/test_traceability.py` and passes on the existing
file (raw run: case C GREEN). Append `(?!\w)` to the `\.py` in `_check_owning_row_tests`.

### R3 — a path-only owning cell passes on bare file existence
Cases D/G (raw runs GREEN): a BUILT row citing only a path is green even when nothing ties the file's
content to the row. Consider requiring the path file to define at least one test, or pair a path with at
least one exact def name.

### R4 — the restructured README dropped the persisted-state probe from its enumeration
Pre-batch, skills/product-prover/README.md's Property-analysis bullet listed "older persisted state meeting
current code" among the stress lenses; the refreshed bullet's probe list ends at "dangling references"
while SKILL.md:302 still carries the Persistence-and-versions bullet. One clause restores the summary's
parity with the skill.

### R5 — bash 3.2 + `set -u`: the summary loop errors on an empty array
`for line in "${SUMMARY_LINES[@]}"` (sync-mirrors.sh:344) is an unbound-variable error on bash 3.2/<4.4
when the array is empty (only reachable with an empty skills/ dir). Harden with
`${SUMMARY_LINES[@]+"${SUMMARY_LINES[@]}"}` if the script is meant to be host-general.

### R6 — M-339's Fact under-describes the cut rule
The Fact says "the story cuts at its first colon"; the script also cuts at a period+space
(`cut_at_outside_paren_break`, `:107-109`). Cosmetic drift between matrix and code.

### R7 — the design-review record's resting state disagrees with the tree
docs/design-review/2026-07-16-2p2p0.md says the loop WAITS on the banner-membership ask and "the class
sentence lands only on the human's word [INV-142]" — yet INV-181's three-member kind sentence and M-340
landed in this same batch `[default]`-tagged, before his word. Either the record should say the default
arm was taken (answered-by-default, ask still open), or the sentence should have held; as written the two
artifacts state different resting states for the same ask.

### R8 — the restructure's trigger fix is not live in sessions: the installed skill copy is stale
`~/.claude/skills/product-prover/SKILL.md` still carries the pre-restructure frontmatter description (the
design-reviewer sentence and [INV-141] in the trigger — verified by diff against the repo copy). The whole
point of the description change is autoload-matcher hygiene, which only the installed copy delivers.
INV-180 names installed skills the habit-held drift-net member; the habit has not run for this landing.
