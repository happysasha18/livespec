## F1 — `orphan_guard` false-reds on a concurrent sibling that LAUNCHES during the window  [real-defect, medium]

One line: the by-deed net raises `AssertionError` when a live pid it did not start appears in the exit census
but was absent at entry — which is exactly a legitimate sibling suite that started its browser mid-window.

Why real: `orphan_guard` diffs `_surviving_owners()` (a global scan of every temp root for `livespec_cdp_*`
dirs with a live `OWNER_PID`) across entry and exit and flags `pid not in before`. It has no record of the
pids *this* suite launched; it infers "leaked by me" from "alive now, not alive at entry." A separate live
process that launches its own harness Chrome after `before` is captured, and is still alive at exit, is
therefore blamed as a surviving orphan. The docstring hedges ("a concurrent sibling run **alive on entry** is
ignored"), but `orphan_guard` is precisely the form the docstrings tell a consumer to "prefer … when a
concurrent sibling suite may be running" — so the recommended concurrency-safe net is non-deterministic under
concurrency, which is the flaky-red INV-155 forbids ("green means deterministic").

Direction: safety-preserving — it OVER-reports (false red), never MISSES a real orphan. So it cannot pass green
while broken; it can fail red while clean. That caps severity at medium, but it is a correctness defect in a
shipped net, not mere hardening.

Concrete failure scenario (reproduced by deed): with no sibling alive on entry, wrap a region in
`orphan_guard()`; during the region a separate process creates a `livespec_cdp_*` dir recording a **live** pid
(a real sibling launching its browser). On exit `orphan_guard` raised
`AssertionError: browser-harness orphans survived the run … pid <live> (…)`. On a one-machine parallel CI
matrix (two suites each wrapping their browser tests in `orphan_guard`, staggered starts), suite A reds because
suite B's browser was born inside A's window and still alive at A's exit — a false teardown-regression report.

Root: the net scopes by "any harness-owned dir with a live pid" (temp-dir census) rather than by "pids this
process actually launched." A scope that records the current process's own launched Chrome pids (e.g. from
`_LIVE_BROWSERS`/a per-run marker) and diffs against *that* set would ignore siblings born mid-window too.
Note this is a design limitation of the entry/exit-diff approach, not a typo; folding it is a small spec/code
item, not a blocker for the delta's stated safety goal.

## Areas that HOLD (no defect found)

**`_temp_roots()` enumeration/dedup/safety — holds.** Enumerates `gettempdir()`, `/tmp`, `/var/tmp`, and every
`/var/folders/*/*/T`; deduped by `realpath`; a missing/unreadable root yields nothing (`os.path.isdir` returns
False on an unstattable path, wrapped in `suppress(OSError)`). By deed it returned ~48 real roots with no raise.
It never returns a root that widens what the sweep reaps — the sweep always filters by `PROFILE_PREFIX`, so only
`livespec_cdp_*` dirs are ever candidates. No path to reaping a non-harness dir.

**Multi-root sweep preserves same-boot / cross-boot / ownerless safety — holds (verified by deed).** On
synthetic dirs: ownerless → kept; same-boot live owner → kept (skipped); same-boot dead owner → reaped;
cross/unknown-boot → rmtree-only, never signalled. The candidate set widened but the per-dir decision is
unchanged, and the decision keys on `_pid_alive` (a machine-global `os.kill(pid,0)` probe), so a live sibling in
ANOTHER workspace is detected identically to one in the current workspace — the cross-workspace live sibling is
never killed or rmtree'd. The `os.path.abspath → realpath` change on `exclude`/`path` is a correctness
improvement (matches the realpath'd roots; handles `/tmp → /private/tmp`); `exclude` is `None` on the only real
call site (`__init__` sweeps before its own `mkdtemp`), so it cannot reap the current run's dir regardless.

**Cross-workspace / shared-root deletion — holds.** The sweep now also scans world-writable `/tmp`, `/var/tmp`
and other users' `/var/folders/*/*/T`. A foreign live run: `os.kill(pid,0)` → `PermissionError` → treated as
alive → skipped. A foreign dead-same-boot dir under a sticky `/tmp`: `rmtree` fails on the sticky bit,
`ignore_errors=True` swallows it — no cross-user deletion. `killpg` fires only when `os.kill(owner,0)` reports
dead, and a dead pid cannot be a live group leader (an emptied process group cannot be rejoined), so the kill
reaches only the original orphan's surviving children — the intended target. No pid-reuse kill hazard.
Minor non-safety note: a same-user crash leftover whose recorded pid gets reused by an unrelated live process
lingers undeleted until that pid frees (skipped as "alive") — a dir-leak edge, safety-preserving, not a defect.

**`SIG_IGN` skip in `_install_teardown_hooks` — holds.** `prev is signal.SIG_IGN` is the correct, complete
guard: `getsignal` returns the singleton `SIG_IGN`, so identity is right. SIGINT and SIGTERM are read and
decided independently per iteration, so differing dispositions (SIGINT = Python's default callable, SIGTERM =
`SIG_DFL` or host-set `SIG_IGN`) are each handled on their own. With `SIG_IGN` filtered before install, the
chained handler's `_prev` can only be a callable, `SIG_DFL`, or `None`; each routes correctly (callable →
chain; else → reset to `SIG_DFL` + re-raise). The in-handler `_prev is SIG_IGN` branch is now dead but
harmless. Respecting a host's deliberate `SIG_IGN` (atexit still carries teardown) is sound.

**`_surviving_owners` / `surviving_orphans` scoping — holds.** Scoped to `PROFILE_PREFIX` dirs carrying an
`OWNER_PID` marker, so the user's own Chrome is never counted; reads live OS state via `_pid_alive`.
`surviving_orphans()` is honestly documented as unsafe under concurrency (defers to `orphan_guard`) — see F1
for `orphan_guard`'s residual mid-window gap. Minor: a leaked dir whose dead owner pid was reused reports as
"surviving" (false), but only matters when teardown killed the process yet failed to rmtree — an edge.

**Spec claims are backed by real code, not docstrings — holds.** The template nets assert on `_stripped_code` /
`_func_code`, which delete the module and per-function docstrings and all comments before the token search
(verified by reading the helpers and by the passing run). Removing a real construct reddens the net even while
its prose survives. INV-157's split (pack string-checks the template; consumer asserts by deed) is honoured.

**CI action bumps — holds.** `actions/checkout@v4→v5` (2 in gates.yml, 1 in stranger-monitor.yml) and
`actions/setup-python@v5→v6` (1 each) are the real Node-24-native majors; grep confirms every occurrence was
bumped uniformly — no `@v4` checkout or `@v5` setup-python straggler, and only these two workflow files use
these actions. Caveat (pre-existing, not introduced here): the repo pins floating major tags rather than commit
SHAs; the bump keeps that existing convention. GitHub-hosted `ubuntu-latest` supports the Node-24 runtime, so
no runner-compat blocker.
