| F3 | Sweep killpg can hit an unrelated process group after a reboot (pid-reuse across the pid namespace reset). Profile dirs live in the macOS per-user TMPDIR (`/var/folders/...`), which persists across reboots. Post-reboot, a recorded owner pid P can be (a) dead as a pid while P is the pgid of an unrelated same-user live group (the leader-exits daemon pattern) → `_pid_alive(P)` says dead → `os.killpg(P, SIGKILL)` kills innocent processes; (b) recycled to a live unrelated process → `_pid_alive` says alive → the leftover dir is skipped forever (permanent leak, the sweep's "cleaned on the NEXT launch" promise silently broken). Within one boot the hazard is closed by POSIX (a pid is not reused while it is a live group's pgid), so the concurrent-sibling safety claim itself holds; the reboot boundary is where it breaks. Fix direction: record boot time (`kern.boottime`) or a launch timestamp alongside the pid and treat a pre-boot marker as "rmtree only, no killpg". | defect (blocks) | headless_harness.py:157–162, 120–131 | **folded** — the sweep records a boot id; a cross-boot marker is rmtree-only (a reboot already killed every process), a same-boot dead owner is killpg'd then removed |
