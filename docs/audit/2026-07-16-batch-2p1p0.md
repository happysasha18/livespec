## F1 — The update watcher is silent in the scenario INV-177 was born from  [defect, high]

One line: a host whose vendored gates pin an old pack version gets NO re-install proposal when the
local pack checkout is itself current — the manifest arm lives inside the "pack update available"
branch, and that branch never runs when installed == remote.

Reproduced by run: manifest pinned to 1.9.0, pack VERSION 2.0.0, remote VERSION 2.0.0 →
`check-pack-update.sh` prints "pack update check: up to date (2.0.0)" and exits before the manifest
is ever read (script exits at the up-to-date check, `scripts/check-pack-update.sh:58-61`; the
INV-177 arm starts at line 67). With remote forced ahead (1.9.0 installed, 2.0.0 remote) the arm
fires correctly — which is the only shape `tests/test_update_watcher.py` exercises
(`remote_version="9.9.9"` in every case).

Why this is the worst finding: row 348's motivation sentence is "a host could run stale gates
forever with a green check", and its Done-when says "an old pin proposes with stale files named."
The ordinary life of this machine is: the pack repo pulls/bumps first, the hosts lag. From that
moment installed == remote every day, and every lagging host's stale pin is invisible forever — the
exact standing state the invariant exists to catch. INV-177's spec sentence ("When the daily check
sees the pack moved past the PIN, it proposes the re-install") is written correctly; the
implementation substituted "moved past the installed checkout" for "moved past the pin."

Recommended fold: hoist the manifest arm out of the update-available branch — compare the pin
against the LOCAL pack VERSION (the same value the per-file diff already reads against), propose
whenever pin < local pack version, regardless of the remote comparison. Add the red test:
installed == remote, old pin → proposal expected.

## F2 — The third net's comment-strip is `#`-only, so the folded evasion is alive in the JS family  [defect, medium]

One line: a JS/TS file that launches Chrome unmuted passes the muted-launch gate when
`--mute-audio` appears in a `//` comment — the same "comment-only mute flag satisfies the mute
check" evasion row 337 declares caught and folded.

Reproduced by run: a `.js` fixture with `// we should add --mute-audio someday` above a real
`puppeteer.launch(...)` with no mute flag → `check-muted-launch.sh` exits 0, OK. The stripper is
`awk '{ sub(/#.*/,"",l) }'` (`guardrails/check-muted-launch.sh:48`) — it strips `#` comments only,
while the gate's own scan set is `*.sh *.py *.js *.ts *.jsx *.tsx *.mjs *.cjs` (line 79), five of
whose members comment with `//` and `/* */`. The red-proven test
(`test_comment_only_mute_still_reds`) uses a Python `#` comment, so the suite proves the fold for
exactly the one comment syntax the stripper handles. Side effect the same line causes: a JS private
field or a `#fragment` in a string is truncated as if it were a comment (no false pass found from
this, but it is the same wrong parser).

Concrete failure: tlvphotos — the very consumer row 337 names, whose harnesses are JS-adjacent
tooling — forks a puppeteer launch, leaves a `// TODO --mute-audio` note, and the gate greens while
sound plays. Row 337's Done-when as literally written ("reds a real unmuted headless-Chrome launch,
passes a muted one and a flag-naming docstring") is MET; the fold-completeness claim in its landing
cell ("the mute check moved onto stripped code") holds only for `#`-commented languages.

Recommended fold: per-extension comment stripping (`//` + `/* */` for the js family, `#` for
sh/py), and a red `.js` fixture beside the existing `.py` one. Keep the honest-boundary sentence;
this is inside the boundary the spec claims, not outside it.

## F3 — The installed Stop hook is a gate living twice with no config-health arm  [composition seam, medium]

One line: INV-175 says "the installed gate is the source gate, held by a config-health check", and
the SAME batch shipped a second installed gate — `hooks/scissors-scan.py` copied to
`~/.claude/hooks/` by `scripts/install-pack-hooks.sh` — that no health check reads.

Mechanics: `check-config-health.sh` iterates exactly `pre-commit pre-push` against `.git/hooks`
(lines 27-38). Nothing in the suite, the push gate, or the update watcher compares the installed
`~/.claude/hooks/scissors-scan.py` against its `hooks/` source (`grep` confirms: no scissors
reference in guardrails/ or test_config_health.py). The installer is idempotent and self-compares —
but only when someone runs it, which is the precise failure INV-175 names: "a stale installed hook
silently under-runs the source's gate list... a gate believed wired stays unenforced." Every future
improvement to the scissors patterns (or the quoted-demo skip, which this batch itself added)
strands the machines that installed before it.

Concrete failure: this machine installed the hook today; next month the pack fixes F4 below in
`hooks/scissors-scan.py`; the suite and push gate stay green forever while the machine runs the old
hook. INV-173 even warns "a hook living only in an installed location has no home to update from,
and that is a defect of this law" — the mirror case (a home whose installed copy is never
re-checked) fell between the two landings.

Recommended fold: a third loop in `check-config-health.sh` — when `$HOME/.claude/hooks/<name>`
exists for a hook shipped in `hooks/`, `cmp` it against source and red with "run
scripts/install-pack-hooks.sh"; absent = skip by name (a machine that never opted in owes nothing).

## F4 — scissors-scan reads only the last assistant transcript event  [defect, low-medium]

One line: a reply split across several assistant events — the ordinary shape of a Claude Code turn
with multiple content blocks or tool calls between text — gets only its final chunk scanned, so a
scissors frame in any earlier paragraph escapes the block.

Reproduced by run: a two-event transcript (event 1 carries "This is a fix, not a workaround.",
event 2 a clean closing line) → exit 0, no block; the same first line alone → blocks.
`last_assistant_text` overwrites `text` per event (`hooks/scissors-scan.py:63-86`) and keeps the
last non-empty chunk. `tests/test_ratchet_kit.py::_write_transcript` writes exactly one event, so
the suite never meets the multi-event shape.

Recommended fold: accumulate every assistant text event after the last non-assistant event (or all
trailing consecutive assistant events) and scan the concatenation; one multi-event red test.

## F5 — "scans every tracked script" over-claims the extension allowlist  [over-claim, low]

One line: INV-157/INV-160/the README say the third net "scans every tracked script", but the repo
scan is `git ls-files '*.sh' '*.py' '*.js' '*.ts' '*.jsx' '*.tsx' '*.mjs' '*.cjs' 'scripts/*'` —
a tracked extensionless launcher (`bin/serve` with a shebang) outside `scripts/` is never read.
The spec's honest-boundary sentence covers bare-shell launches, runtime-built arg lists, and
non-Chrome drivers; it does not cover the extension gap, which is a scanner scope hole rather than
a signal-detection boundary. (Also noted: the directory-mode file set adds `*.txt *.md`, so the
gate's two modes scan different populations.) Fold: pick up extensionless tracked files whose first
line is a shebang, or add the extension boundary to the spec's boundary sentence so the claim and
the scan agree.

## F6 — Two landings minted the same gate letter "m"  [collision, low]

One line: row 346 wired config-health as the pack pre-push's "gate m"; row 345's installer appends
"gate m — ratchet caps" to a HOST's pre-push when absent. The README tells a host to copy the
pack's pre-push as its starting point — a host that does so and then runs `install-ratchet.sh`
carries two gates named m in the one enumeration row 352 just declared kept-current. Fold: letter
the ratchet gate "n" in the installer (one string), and let the README's a–m line grow with it.

## F7 — A stale quantifier survives the batch that shipped the quantifier law  [recommendation, low]

One line: `skills/live-spec-base/SKILL.md:8` titles "one rulebook, eight working skills" while the
same file's description enumerates nine (spec-author, product-prover, design-reviewer,
build-pipeline, test-author, communicator, feedback-intake, feedback-collector, publish) and
`skills/` holds ten dirs minus the base. Pre-existing debt (origin/main carries it), so no landing
regressed it — but the batch shipped INV-170 (quantifier re-verify), INV-176 (a law that earns a
gate scans retroactively), and row 352 fixed this exact class in guardrails/README, while INV-170
itself got no retroactive arm, so the pack's most-loaded doc keeps a falsified count. Fold: run the
quantifier re-verify once backward over the pack's own doc set (the catch-up shape INV-176 already
prescribes for gates) and fix the count; the loader note in ~/.claude/CLAUDE.md repeats "eight" and
wants the same sweep.

## F8 — Single-quoted demos are not skipped by the scissors hook  [recommendation, low]

One line: `'X, not Y'` in a reply blocks (reproduced by run) while `«X, not Y»`, `"X, not Y"`, and
backticked demos pass; INV-173's sentence ("text inside quotation marks... names a pattern rather
than using it") and M-328's "quotes" read naturally to include single quotes, and the hook's own
rewrite instruction demonstrates the frame in single quotes — safe only because `stop_hook_active`
suppresses the second pass. Fold: add `'...'` spans to `_strip_quoted_demos`, or name the three
skipped quote types in the spec and matrix so the claim matches the code (apostrophes make the
regex touchier — `[^']` bounded to no-space-adjacent openings, or the doc fix alone).
