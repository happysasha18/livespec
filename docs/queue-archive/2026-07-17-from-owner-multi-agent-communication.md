fine for wishes, but agents should be able to COLLABORATE — the promoter agent needs the website's
data (the site run by tlvphotos / exhibition-engine) to build campaigns, analyze, and propose A/B

sees what works. Agents may share one machine and may later be remote. Thought through by the Fable

## What already exists (build on it, never beside it)

- **The wish door (inbox/)**: one-shot, git-atomic, human-auditable requests to CHANGE something.
  Right grain for "site, please add coupon support"; wrong grain for "give me this week's funnel
  numbers" — data is recurring, structured, and read-only.
- **The window-isolation law**: one window = one project tree; a foreign tree is READ-ONLY. The
  2026-07-17 memory note already fixes the pair: site is read-only to promoter, they talk via
  inbox wishes both ways; the deeper protocol was left TBD. This wish is that TBD.
- **The remote-seat law**: a remote seat reaches a repo only through git. So git is the universal
  transport; a shared machine is only a fast path, never the design.

## The design as thought (four separable stories)

1. **The data-feed contract (producer side).** The site PUBLISHES a versioned, machine-readable
   feed (a JSON artifact in a known repo path, regenerated by its own scheduled task or on deploy).
   The contract is a spec surface in the producer's own PRODUCT_SPEC: every metric names its
   meaning, measurement window, aggregation, and source; the feed carries a generation timestamp
   and the contract version. Proven and tested on the producer's side like any surface.

2. **The consumer pin + freshness discipline.** The consumer (promoter) pins a contract version and
   reads the feed read-only (same machine: direct read of the producer's tree, already legal;
   remote: git pull). Before any analysis the consumer checks the feed's age against a declared
   staleness bound; stale data is named, never silently analyzed. A compatibility test lives on the
   consumer side and reds when the pinned contract version and the feed's version diverge.

3. **The experiment lifecycle.** A promoter proposal (A/B test, coupon, funnel change) enters
   through the site's wish door carrying an EXPERIMENT ID and its success metric (named in contract
   terms). The site's own pipeline decides and builds it (the owner's gates intact — a product
   change on his portfolio is his word). Running experiments tag their metrics with the ID; the
   feed carries per-experiment outcomes; the promoter closes the loop (kept / rolled back /
   inconclusive — and the curate-hard law applies: inconclusive is reported flat). The lifecycle
   (proposed → decided → running → measured → concluded) is a spec'd protocol, each transition
   owned by a named side, auditable in git.

4. **The report-back cadence.** The producer's scheduled task regenerates the feed on a declared
   cadence ([default] daily, set on the owner's word); the consumer's standing loop reads it and
   digests "what worked" for the owner. Cadence, like any budget, names its watcher (INV-41 now
   demands this): a feed older than its cadence reds the consumer's freshness check.

**The general case (his own 2026-07-17 law: lift the particular to the general).** The
promoter↔site pair is the WORKED INSTANCE. The general law: any two project agents communicate
through exactly two declared channels — the wish door (requests to change, one-shot, swept by the
owner) and published data contracts (recurring reads, versioned, freshness-watched). No third
back-channel; a shared machine changes transport speed, never the contract. This slots into the
pack as a base-rule growth + one new skill (the channel contract: authoring, pinning, the
compatibility test, the experiment lifecycle) + a template.

## The second half (the owner's word, 2026-07-17 ~11:32, added same day)

Agents must KNOW each other and negotiate; four more stories join the movement:

5. **The agent registry.** Each agent publishes a short self-description in a known place: its
   name, mission, the zones it owns, the contracts it publishes, its wish-door address. Any agent
   reads the registry before acting, so "who does what" is a lookup and never tribal memory.

6. **The non-duplication rule.** An agent that needs a capability another agent's zone owns takes
   the road of a wish or a contract read; building a local copy of a neighbour's capability is the
   violation the registry exists to prevent. (His example: tlvphotos must not grow what another
   agent already does, yet sometimes data or access must be shared — that sharing is exactly the
   two channels above.)

7. **Agent birth as a first-class act.** When a reusable capability is owned by nobody (or has
   outgrown its host), any agent may PROPOSE a new agent; the owner ratifies the birth. The worked
   case: analytics lives inside tlvphotos/exhibition-engine today and may become its own agent
   later — so the contract must outlive the migration (the feed's address and version move, the
   consumer's pin survives, the registry records the new owner).

8. **The skill/agent boundary, stated.** A skill is a capability any window loads: stateless, no
   tree of its own, no mission, dies with the session. An agent is a project window: its own tree,
   queue, gates, contracts, standing mission, and a seat in the registry. The promotion road
   skill→agent (and the demotion road back) is the same birth act as story 7, and the registry is
   where the boundary is visible. The undecided grain per capability (this analytics: skill inside
   the engine, or agent of its own?) is an owner call the registry records with its date.

## Trigger note (the owner's question, 2026-07-17 ~11:41)

Data flows on the AGENTS' own clocks, never on the owner's builds: the producer's scheduled task
regenerates the feed by itself (a deploy may also refresh it, as a bonus, never as the trigger),
and the consumer reads on its own demand behind the freshness check. The later rung, seeded here:
event-driven waking — the producer notifies the consumer on a feed change (a repo push the
consumer's monitor watches is the git-native form) so a standing consumer loop can react instead
of polling. Polling at the declared cadence is the honest v1; the wake-on-change rung upgrades it
without touching the contract.
