What setting up looks like

You answer one question and get told one thing. That is the whole setup — here it is start to finish:

reads your project→ asks you 1 question→ tells you 1 thing→ shows this summary→ asks the rest later, one at a time
  • 1It reads through your project on its own. It will never ask you about anything it can find in your files.
  • 2It asks you one question, to check what it found: “This looks like a static website — is that right?” You say yes, or you correct it.
  • 3It tells you one thing, no answer needed: how thorough it will be on each change, and the exact words that make it move faster when you are short on time.
  • 4Then this summary appears, once. Any time later, type “what can I customize?” and it is built again from your current settings and shown to you.
  • 5A few other choices are not decided now. It asks each of them later, at the first moment it cannot go further without your answer — one question at a time. For example: where your code gets pushed, the first time you push it; and the first time a change has a visual result, it opens the drawn page in front of you and asks whether the look is right before it builds any further.

live-spec is now set up on this project

From here on, every change you ask for runs the same way: first it writes down exactly what you will get, checks that description for gaps, turns it into tests — and only then builds it and shows you the result. The settings below decide how that feels day to day: what language things are written in, how often it stops to ask you, and how thorough it is on each change. Every setting already has a value that works. You can change any of them, any time, by saying so in the chat.

What setup settled about this project

One you answered, one you were told. These two shape everything else, which is why they are the only ones handled up front.

What kind of project this is
skill pack — a bundle of working rules and tools for an AI agent

What it changes for you: this decides what a change has to show you before it counts as finished. For a bundle like this, every rule the pack promises must have code that enforces it and a test proving that code works — you will see both before the change is marked done. For a website instead, “finished” would mean a test that drives the page in a browser and a fresh screenshot of it for you to look at.

to change: name a different kind — book, backend service, static website, full app, command-line tool, agent skill pack, or describe your own.
How thorough it is on each change
every step, every time

What it changes for you: right now each change you ask for gets the whole treatment — it writes down what you will get, checks that for gaps, writes the tests, then builds it. When a deadline is tight, tell it so: on small changes it will then run fewer of those checks and save the rest for the end, and it names out loud each step it skipped so you always know what was traded. Whatever the pace, some things are never skipped: tests are written before the code, nothing is deleted without telling you, and it stops to ask you before anything that cannot be undone.

to change: say “we’re in a hurry” or “take your time, be thorough” whenever the pace should shift.

How often it stops to ask you
decides small things itself

What it changes for you: when a choice has an obvious answer, it picks it, keeps working, and tells you what it picked together with the result — so you read one short summary instead of answering ten questions. It still stops and asks you about matters of taste, about anything to do with how things look, and about anything that cannot be undone.

Two rules this project sets for itself

This repository is public — its readers can see the method itself — so before every push it re-checks the whole written spec for gaps. On most projects that full check runs only at version milestones, not on every push.
