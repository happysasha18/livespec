# Back-described code descriptions — FORM sample for Alexander's accept (2026-07-20)

This is a sample, not the migration. Before the full pass back-describes every registered code (the one-pass major-release chapter INV-239 calls for), read these fifteen and accept the FORM: does each one-sentence description read clear to someone who does not build the method, and does it hold the quality bar (say what the item does and the problem it solves; show the whole class when the rule is a class; name its key term in plain words; use the accurate actor and object)? The spread crosses the kinds on purpose, and several codes were picked because their meaning is not obvious, so the sample stresses the bar rather than flattering it.

## Notes

### Codes whose meaning needed a judgment call — flag for your read

- **ACT-3 — "the ledger walk."** The index says every brief carries "the ledger walk + the clock." The ledger walk is the searching-the-records step a worker runs (problem ledger, prior answers), so I rendered it as "the records walk." If you read "ledger" as strictly the problem-ledger (E-24) rather than the wider records sweep, the plain word should change. Worth your eye.
- **M-1 — "gate list."** The description names "the full gate list," which is what the index says, but the exact contents of that list live across other clauses and some legs are still marked target. The description stays faithful to the index without pinning a list that is not fully shipped. If you want the milestone description to enumerate the gates, that is a decision, not a back-description.
- **E-35 — mild self-reference.** The named reference is the very thing this whole migration builds, so describing it uses the concept it defines. The line reads cleanly on its own, but if you would rather the canonical E-35 description avoid the word "reference" describing "reference," say so and it can be reworded once for the field's one home.

### Where the bar's four rules pulled against each other

- **The class laws (INV-153, INV-98, INV-218) put rule (b) against rule (c).** Stating the class in the abstract ("everything routes to its governing home", "every process converges") is the most faithful to (b) show-the-whole-class, but the abstract sentence reaches for exactly the undefined abstract word rule (c) forbids. I resolved it by naming the class and then listing its concrete members inline (for INV-153: a request, a property, a deferred row, a cross-project message), so the sentence reads as its class while staying plain. This is the main form question for your accept: is the "class named, then its members listed inline" shape the right register, or too long? It is the pattern the whole class-law family will follow.
- **T-19 and M-1 put rule (a) against sentence length.** Saying what the item does AND the problem it solves, for a multi-rung ladder or a five-step gate, wants a longer sentence than the simple invariants (INV-1) need. I kept one sentence each but leaned on a parenthetical to carry the members. If you want a hard length cap, the class laws and the ladders are where it will bite first.
